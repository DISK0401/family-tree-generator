# NDLOCR / NDLOCR-Lite は手書き戸籍(明治〜昭和)の読み取りに使えるか — 技術評価と推奨パイプライン

## TL;DR
- **結論:NDLOCR および NDLOCR-Lite は、明治19年式〜昭和23年式の手書き・毛筆・縦書き戸籍の本文読み取りには「実質的に使えない」。** これらは明治以降の「活字」刊行物向けに学習されたOCRであり、手書き行政帳票とのドメインギャップが決定的。独立ベンチマーク(逆瀬川氏, 2026年3月)では、手書き文字に対しNDLOCR v2はNLS 0.064で全23モデル中最下位、NDLOCR-Liteも0.271で下位に沈み、Gemini 3.1 Pro(0.924)やGoogle Cloud Vision(0.820)に大差で敗れた。
- **手書き縦書き戸籍の最有力手段はマルチモーダルLLM(特にGeminiファミリー)であり、前回調査のGoogle Cloud Vision/Document AI + LLM構成の「LLM」を主役に据えるのが最善。** NDLOCRを下読みエンジンに据える構成は精度を下げるため非推奨。旧字体・くずし字は「NDL古典籍OCR」の守備範囲だが、これも江戸期の版本(木版印刷)中心で毛筆行政文書には最適化されていない。
- **個人開発の現実解:有償目玉機能は「Gemini等のLLMでOCR+構造化 → ルール検証 → 人間校正」パイプラインで構築し、NDLOCRはコスト・精度いずれの面でも組み込む必然性がない。** 戸籍は生存親族の機微な個人情報を含むため、クラウドLLM利用時はデータ学習オプトアウト付きの法人/API経由を必須とし、オンプレ要件が出た場合の代替としてYomiToku(自己ホスト)を評価する。

## Key Findings

1. **NDLOCRの本質は「活字資料OCR」である。** 国立国会図書館(NDL)次世代システム開発研究室のNDLOCRは、令和3年度・令和4年度の事業で「明治期以降の活字資料」を対象に開発された。令和3年度の大規模テキスト化事業はLINE株式会社に委託して実施され、令和2年12月時点で提供していた図書・雑誌等の活字資料 約247万点(約2.2億画像)を対象とした。学習・適用の中心が刊行物であり、手書き行政文書とはドメインが根本的に異なる。

2. **手書き対応は「実験的」レベルにとどまる。** NDLOCR-Lite(2026年2月24日公開)は公式に「NDLOCRが不得意としていた英文や手書き文字等についても実験的に対応しています」と明記(NDLラボ「NDLOCR-Liteの公開について」)。公式CERは学習用ベンチマークJaWildTextの手書き1,065画像で全体平均0.268(縦書き0.279、横書き0.264)であり、これは切り出された比較的読みやすい現代手書きシーンテキストに対する値で、毛筆・旧字体・地紋つき戸籍の値ではない。

3. **独立ベンチマークで戸籍類似タスク(日本語手書き)の実力が露呈。** 逆瀬川氏の23モデル比較(2026年3月、手書きメモ6画像・Hungarian NLS主指標、評価コードは GitHub nyosegawa/ocr-comparison)では、**NDLOCR v2はNLS 0.064・CER 0.958で23モデル中最下位、NDLOCR-Liteは0.271・CER 0.915で21位**。出力は文字化け(〓)や欠落が大半だった。同記事は「NDLOCRは活字には強いが手書きノートは守備範囲外」と結論。一方Gemini 3.1 Pro 0.924、Google Cloud Vision 0.820、YomiToku(OSS最上位)0.770。

4. **くずし字は「NDL古典籍OCR」の担当だが、戸籍とはズレる。** NDL古典籍OCR/同Lite(RTMDet+PARSeq)は江戸期以前の和古書・漢籍向け。NDL自身「くずし字や漢籍も読み取れる場合があるが、本格的なテキスト化にはNDL古典籍OCRを推奨」。しかしCODH由来の学習データは版本(木版印刷)中心で、明治期の毛筆行政文書(古文書・古記録)は「あまり含まれていない」とCODHが明言。

5. **CODHのKuroNet/RURI(瑠璃)・みをも版本中心で、戸籍向きではない。** CODH公式は「こうした問題が存在するため、くずし字OCRは未だ実用的なレベルに達していないのが現状です」「手書き文字は印刷文字より変化が大きく容易でない」と明言。RURIは江戸期木版本で最高性能、手書き古文書は対象拡大の途上。みをのライセンスはCC BY-SA(非営利・研究推奨)で商用サービス組み込みには不向き。

6. **戸籍手書き読み取りの本命はマルチモーダルLLM。** 複数の実証で、Geminiファミリーが日本語手書き・縦書き・旧字体・漢数字に対し最良。逆瀬川ベンチでGemini 3.1 Proが首位、Flash Liteも0.899で僅差かつ5倍高速。ただしLLMは「読めない文字を創作(ハルシネーション)」するリスクがあり、GPT-4.1は縦書き旧字文学で大規模な幻覚が報告された(Geminiは機械的誤字中心で相対的に安全)。

7. **専業ベンダーですら手書き戸籍は難物。** TOPPAN(凸版)は明治〜昭和初期の近代手書き文字AI-OCRを開発(2022年)。同社の「ふみのはゼミ」ではAI-OCRの目標精度を「版本の読み取りで90%」と明言し、古文書カメラ搭載モデルは「解読率90%」を謳う一方、手書きはこれを下回る(業界・実証で手書き60〜80%水準)。TOPPAN Edgeとあいち銀行は2026年3月11日の共同リリースで「国内初となる手書き戸籍謄本の読み取りが可能なAI-OCRサービスの実用化を目指します」とし、技術基盤はTOPPANのくずし字古文書解読サービスに立脚。ただし精度は未公表・実用化を「目指す」段階(将来予測的表現)。既存の家系図/相続自動作成サービス(サムポローニア「AI相続ミツローくん」等)の戸籍AI-OCRは解析対象が「平成以降」に限定されるものが多い。

## Details

### 1. NDLOCR各バージョンの整理(ユーザーの用語混同を訂正)
「NDLOCR ver.3 = NDLOCR-Lite」は誤解。実際にはNDLには2系統×各Lite版がある。

| 名称 | 対象資料 | アーキテクチャ | 動作要件 | ライセンス | 公開時期 | 入手先 |
|---|---|---|---|---|---|---|
| NDLOCR ver.1 | 明治以降の活字 | TrOCR等(外部委託, 令和3年度) | GPU必須 | CC BY 4.0 | 2022年(〜2023/6) | ndlocr_cli -b ver.1 |
| NDLOCR ver.2(現行) | 明治以降の活字 | 行内/行間スコア改善版(令和4年度追加開発) | GPU必須 | CC BY 4.0 | 2023年〜 | github.com/ndl-lab/ndlocr_cli |
| **NDLOCR-Lite(v1.x)** | 図書・雑誌の活字(手書き実験対応) | **DEIMv2(レイアウト)+PARSeq(文字認識)+読み順整序、ONNX変換** | **GPU不要・CPU可、RAM約1GB** | CC BY 4.0 | 2026年2月24日 | github.com/ndl-lab/ndlocr-lite |
| NDL古典籍OCR ver.3 | 江戸期以前の和古書・漢籍(くずし字) | 独自(内製) | GPU必須 | CC BY 4.0 | 2022年〜 | ndlkotenocr_cli |
| NDL古典籍OCR-Lite | 同上(くずし字) | RTMDet+PARSeq、ONNX | GPU不要・CPU可 | CC BY 4.0 | 2024年11月26日 | ndlkotenocr-lite |

- NDLOCR-Liteは3モジュール構成(レイアウト認識/文字列認識/読み順整序)。PyTorch学習後にONNX変換しCPU推論を実現。Apple Silicon Macで1画像1〜5秒、メモリ約1GBで動作。出力はTXT/JSON/XML/TEI/透明テキストPDF。CLI(Python 3.10+)とGUI版あり。GPU利用も`--device cuda`でベータ対応。
- ライセンスはCC BY 4.0で商用利用可(クレジット表示条件)。ただし内部依存ライブラリ(Apache-2.0/MIT/BSD)の条項にも従う必要がある。**戸籍OCR出力を商用サービスに組み込むこと自体はライセンス上は問題ない**——問題は精度。

### 2. 戸籍というドキュメントの難しさ
現在取得可能な戸籍は明治19年式(1886)〜平成6年式の5様式。平成6年式(電算化)のみ活字・横書き・A4。それ以前(明治19年式・明治31年式・大正4年式・昭和23年式)はすべて **手書き(毛筆/ペン)・縦書き・旧字体・漢数字(壱弐参拾)・罫線帳票** で、戸主中心に多人数を1戸籍へ記載する複雑レイアウト。加えて実務で扱うのは原本でなく謄写・全部事項証明書のコピーで、地紋・滲み・訂正線(朱線)・訂正印が混じる。これはOCRにとって最悪クラスの条件が全て揃っている。

戸籍特有課題×NDLOCRの適性:
- (a) 罫線・枠のある帳票 → NDLOCRのレイアウト認識は図書の段組・本文向けで、帳票セル構造は想定外。図表混在で認識が総崩れになる既知の弱点あり。
- (b) 毛筆・行書の手書き → **最大の壁。学習データが活字のため実質非対応。**
- (c) 旧字体・異体字 → NDLOCRは字種表を持つが手書き字形の崩れに弱い。
- (d) 朱書き・訂正印・付箋 → 学習外ノイズで誤認識・欠落を誘発。
- (e) コピーの地紋・劣化 → 前処理なしでは致命的。

### 3. ドメインギャップの定量的裏付け
逆瀬川氏(2026-03)の日本語手書き比較(Hungarian NLS主指標、手書きメモ6画像、CER併記):

| モデル | 種別 | NLS(高いほど良) | CER(低いほど良) |
|---|---|---|---|
| Gemini 3.1 Pro Preview | API | 0.924 | 0.205 |
| Gemini 3.1 Flash Lite | API | 0.899 | 0.207 |
| Claude 4.6 Opus | API | 0.897 | 0.225 |
| Azure AI Vision | API | 0.830 | 0.332 |
| Google Cloud Vision | API | 0.820 | 0.509 |
| YomiToku(OSS最上位) | Modal | 0.770 | 0.400 |
| **NDLOCR-Lite** | Modal(CPU) | **0.271** | **0.915** |
| **NDLOCR v2** | Modal(A10G) | **0.064** | **0.958** |

戸籍の手書き本文はこの「手書きノート」より難しい(毛筆・旧字・地紋)ため、NDLOCR系の実効精度はこれ以下と見るのが妥当。**下読みエンジンとして使っても、後段LLM補正の入力を汚染するだけで逆効果。**

### 4. マルチモーダルLLMとの比較(手書き縦書き戸籍における相対評価)
- **強み(LLM):** 文脈推論で崩し字・似た字形を補完、縦書き・旧字体・漢数字対応、OCRと構造化(氏名/続柄/生年月日/本籍の抽出)をワンショットで実行可能。Geminiは日本語手書きで首位級かつ安価。
- **弱み(LLM):** ハルシネーション(実在しない人名・情景の創作)。特に固有名詞や訂正箇所で危険。金額・日付・人名は必ず人間校正が必要。GPT系は縦書き日本語でGeminiに劣後する報告が複数。
- **NDLOCRの相対的強み:** ほぼ皆無(手書き戸籍において)。唯一の利点はローカル完結(個人情報を外部送信しない)だが、これはYomiTokuやローカルLLMでも代替可能。
- **汎用OCR(Google Cloud Vision/Document AI)の位置づけ:** 平成6年式(活字横書き)には十分。手書き旧様式ではNLS 0.820とLLMに劣るが、高速(2.2秒)・安定で「読める字は堅実に読む」。

### 5. 実装・コスト(個人開発視点)
- **LLM API方式(推奨):** Gemini APIは画像1枚が1024×1024で約1,290トークン。Flash系(入力$0.30〜0.50/1Mトークン、出力$2.50〜3.00/1M)なら **1ページ数円以下**。月数千ページでも数千円規模。サーバ運用不要、スケール自在。前処理(傾き補正・地紋除去・二値化)をローカルで行いLLMに渡すと精度が上がる。
- **自己ホスト(NDLOCR-Lite):** CPUで動くのは魅力だが手書き戸籍では精度が出ないため投資対効果が低い。もし活字(平成6年式)専用の下処理に使うなら、CPUコンテナ(Docker)で安価に常駐可能。
- **自己ホスト(YomiToku等)でGPUが要る場合:** サーバレスGPUが現実的。Modalは秒課金(A100 40GB約$0.001/秒、$30/月の無料クレジット)、RunPod ServerlessはRTX4090約$0.34〜0.58/時・A100約$1.39/時と安価。スパイク型トラフィックはModal(秒課金・高速コールドスタート)、定常負荷はRunPod常駐ワーカーが有利。個人開発の低頻度アクセスならスケールtoゼロのサーバレスGPUが最適。
- **ライセンス面:** NDLOCR系はCC BY 4.0で商用組込可。YomiTokuはCC BY-NC-SA 4.0で **商用は別途商用ライセンス(MLism社/AWS Marketplace版)が必要**。みを/くずし字データセットはCC BY-SA系で商用制約あり。GeminiはAPI利用規約に従い、個人情報は学習不使用の設定/プランを選択。

### 6. 推奨パイプライン(前回調査の構成にNDLOCRをどう位置づけるか)
前回結論(平成6年式=汎用OCR+LLM)を土台に、様式で分岐:

1. **様式判定・前処理:** スキャン画像を受領 → 傾き補正・地紋除去・コントラスト強調(OpenCV) → 平成6年式(活字横書き)か旧様式(手書き縦書き)かを判定。
2. **平成6年式ルート:** Google Cloud Vision/Document AI もしくはGemini でOCR → LLMで構造化。前回調査どおり実用可能。
3. **旧様式(手書き)ルート:** **Gemini(3.x Pro/Flash)をOCR兼構造化エンジンに** → 「氏名/続柄/父母/生年月日/本籍/事項」をJSONスキーマで抽出、旧字体は保持指示、読めない箇所は`(判読不能)`で明示させる(創作抑制)。
4. **検証層:** 日付妥当性・続柄整合・年号整合をルールで独立検証 → 確信度が閾値未満は「要確認」フラグ。
5. **人間校正(必須):** 有償機能でも「AI下書き→人が確定」を前提に。誤読・幻覚を残したまま家系図確定は法的にも危険。
6. **NDLOCRの位置づけ:** **本パイプラインに組み込む必然性はない。** ローカル完結が絶対要件になった場合のみ、活字部分の補助やYomiToku/ローカルVLMの比較対象として検討。

### 7. 検証実験の進め方(手元サンプルでの精度検証)
- **正解データ作成:** 手元の戸籍サンプル(様式別に各5〜10通)を専門家/自分で翻刻し、氏名・続柄・生年月日など「抽出したいフィールド単位」の正解を作る(全文CERより実務に近い)。
- **評価指標:** 全文CERは読み順で不当に悪化するため、逆瀬川氏方式(領域単位のマッチング=Hungarian NLS)やフィールド単位の完全一致率/F1を併用。
- **比較対象:** ①Gemini(Pro/Flash) ②Google Cloud Vision+LLM補正 ③NDLOCR-Lite(参考・下限確認) ④YomiToku ⑤(くずし字が強い場合)NDL古典籍OCR-Lite。同一画像・同一プロンプト・同一スキーマで統一。
- **前処理アブレーション:** 地紋除去・二値化・解像度(短辺720px以上推奨)の有無で精度差を測る。
- **ハルシネーション検査:** LLM出力に「原本にない人名/日付」が混入していないか、正解と突合して幻覚率を必ず測る。
- **様式別の合否基準:** 平成6年式=フィールド一致95%+で自動化可。手書き旧様式=まず「人が校正する前提の下書き」品質(フィールド一致70〜80%目安、TOPPAN手書き目標と同水準)を到達目標に設定し、それ未満なら人手翻刻併用に切替。

## Recommendations
1. **即時(PoC):** 手元の各様式サンプルでGemini(3.x Flash→Proの順)を第一候補に検証。プロンプトで「旧字体保持」「判読不能は明示」「JSON構造化」を指定。ベンチ上の優位と低コスト($数円/ページ)から、ここが本命。
2. **NDLOCR/NDLOCR-Liteは「下限確認用の参考」に格下げ。** 手書き戸籍の本読みには採用しない。活字の平成6年式にはGoogle Cloud Vision/Document AIかGeminiで十分。
3. **プライバシー設計を先に固める:** 戸籍は生存親族の個人情報を含むため、クラウドLLMはデータ学習不使用(オプトアウト/法人API)を必須化。利用規約・プライバシーポリシーに明記。オンプレ要求が強い顧客層が見込まれるなら、YomiToku商用ライセンスかローカルVLMをオプションとして早期に技術検証。
4. **有償機能は「AI下書き+人間確定」を前提に設計。** 全自動確定は精度・法的リスクの両面で時期尚早。確信度フラグと差分ハイライトで校正コストを最小化。
5. **閾値と切替基準:** 手書き様式でフィールド一致率が継続的に80%を超え、幻覚率が実用域(例:重要フィールドで1%未満)に収まれば自動化比率を上げる。それ未満なら人手翻刻を併用。TOPPAN Edge×あいち銀行の手書き戸籍AI-OCR実用化やGeminiの世代更新は四半期ごとに再評価。

## Caveats
- 逆瀬川ベンチは手書き「メモ」6画像の小規模評価で、戸籍そのものではない。傾向は明確だが、戸籍固有画像での再検証が必須。
- NDLOCR-Liteは2026年4月のv1.2で手書き・タイプライター文字を強化中であり、将来の改善余地はある。ただし学習の主眼は活字であり、短期に戸籍対応が実用化する見込みは低い。
- 「手書き戸籍を読む名指しのOCRツール×測定済みCER」を公開した一次情報は確認できなかった。手書き系の具体数値はTOPPANの目標値(版本90%、手書きはそれを下回る60〜80%水準)と、非日本語手書きに対するGeminiのCER(ある報告で1.67%)が最も近い代理指標にとどまる。TOPPAN Edge×あいち銀行の戸籍AI-OCRは2026年3月時点で「実用化を目指す」段階であり精度未公表(将来予測的表現に注意)。
- Gemini等のモデル性能・価格は更新が速く、本評価は2026年前半時点のスナップショット。導入前に最新モデル・料金・規約を再確認すること。
- LLMのハルシネーションは戸籍のような固有名詞主体の文書で特に危険。人間校正を外す判断は本調査の推奨範囲外。
