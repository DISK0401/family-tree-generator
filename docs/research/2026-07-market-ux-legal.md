# 日本市場向けWeb家系図作成サービス 個人開発のための包括的調査レポート

## TL;DR
- **戦うべき土俵は「UX」と「戸籍からの自動生成体験」の2点に集約すべき。** 国内最大手の「みんなの家系図 by 名字由来net」（公式表記「登録数1,500,000人突破」「名字・家系図業界唯一のプライバシーマーク取得企業が運営」）が市場を押さえるが、App Storeレビュー等では「使い方のイントロダクションがない」「西暦入力がちょびっと面倒」「非公開設定にすると保存できない」といった操作性の不満が散見される。無料クライアント完結型でUXを磨き、有償の戸籍自動生成で差別化する構想は妥当。
- **有償の目玉「戸籍スキャン→自動家系図」は、平成6年式（活字・横書き）なら実現可能だが、明治〜昭和の手書き・縦書き・旧字体戸籍は現時点で最難関。** サムポローニア「AI相続ミツローくん」は「すでに横書き・一部縦書き戸籍のAI-OCR読み取りを実用化」、手書き戸籍はTOPPANエッジ×あいち銀行が「国内初となる手書き戸籍謄本の読み取りが可能なAI-OCRサービスの実用化を目指す」段階（2026年3月発表、将来形）。個人開発では「平成戸籍＋LLM構造化」から始め、古い戸籍は人間補正前提のハイブリッド設計が現実的。
- **法的には、家系図作成そのものは最高裁判決（平成22年12月20日、第一小法廷）で行政書士の独占業務ではないと確定しており、個人でも有償提供は可能。** ただし戸籍情報は要配慮情報に準じる慎重な扱いが必要で、有償版を出す時点で「個人情報取扱事業者」としてプライバシーポリシー整備・安全管理措置・漏えい時の報告義務が発生する。戸籍の取得代行には踏み込まず「利用者が自ら取得した戸籍を処理する」設計に限定すべき。

## Key Findings

### 1. 競合とUXの不満点
- 国内市場は「みんなの家系図 by 名字由来net」（Recstu社のアプリ、公式サイトminnano-kakeizu.netは「登録数1,500,000人突破」「名字・家系図業界唯一のプライバシーマーク取得企業が運営」と明記）が事実上の最大手。プレミアム月額240円、3年一括8,000円で永年会員。無料は50名まで登録可、プレミアムで51名以上。
- 買い切り型ソフト（「つくれる家系図」約4,000円、「らくらく家系図作成」「家系図のススメ」2,990円前後、「親戚まっぷ9」360円等）。Windows中心でデータ移行・共有が弱く、和暦非対応（西暦換算必須）のものがある。
- 海外勢はAncestry（サブスク、記録DB膨大、日本語未対応）、MyHeritage（GEDCOM対応・日本語UI有、DNA販売）、FamilySearch（無料・非営利だが故人情報が共有前提）、Gramps（無料OSS、玄人向け）。いずれも日本の戸籍・縦書き・家制度にローカライズされていない。
- ブラウザ型の新興ツール「xGrapher」がインストール不要・無料・モダンUI・家系図専用テンプレートを訴求。
- 共通の不満：①線・レイアウトが崩れる/自動整列が弱い（Excel等は人物追加のたびに線を引き直す苦痛） ②印刷すると画像が荒い ③サービス終了時のデータ消失/移行の不安 ④操作説明・オンボーディング不足 ⑤和暦/旧字体入力の煩雑さ ⑥複雑な家族関係（養子縁組・再婚・複数配偶者）が描きにくい。
- 行政書士等の代行市場：1家系（父方のみ）3〜10万円、全家系16万円前後、装丁込みで数十万円。最高裁事件では被告が家系図6通で計約90万円（別報道では3通で計約33万円＝1通約11万円）を受領しており、実勢価格の裏付けとなる。家系図作成は最高裁判決で独占業務でないと確定し、参入は自由。

### 2. 戸籍OCRの技術的実現性
- 戸籍様式は明治19年式・明治31年式・大正4年式・昭和23年式（すべて手書き・縦書き）と平成6年式（活字・横書き・コンピュータ化、A4横書き）に大別。平成6年式が最もOCR容易で、古いものほど筆書き・くずし字・旧字体・変体仮名で難易度が跳ね上がる。分岐点は「平成6年式か否か（＝活字横書きか手書き縦書きか）」。
- **サムポローニア「AI相続ミツローくん」**（エスクロー・エージェント・ジャパングループ）は2026年4月16日発表（PR TIMES）で「すでに横書き・一部縦書き戸籍のAI-OCR読み取りを実用化しているだけでなく、法定相続人の自動判定までをワンストップで実行」と明記。富士通の金融機関向け「FinSnaviCloud」に搭載済み。郡山市を含む実証（2025年9月〜2026年3月）では「従来、手作業で平均60分かかっていた1件あたりの相続関係説明図作成時間が平均10分へと短縮され、50分の業務時間削減を実現」。計12自治体が2026年度（令和8年度）導入決定、料金は年額24万円〜。特許第6283343号等を保有。
- **TOPPANエッジ×あいち銀行**は2026年3月11日共同リリースで「国内初となる手書き戸籍謄本の読み取りが可能なAI-OCRサービスの実用化を目指します」と発表。TOPPANグループの「くずし字を対象とした古文書解読サービスの技術」を基盤とする協業で、完成品の実測精度は未公表の将来形。
- くずし字OCR「KuroNet/RURI」（CODH）は公称精度85〜90%だが、実利用者の体感は「7割程度」との報告もある。戸籍そのものの公開実測CERは存在しない。
- 汎用マルチモーダルLLM（GPT-4o、Gemini 2.5、Claude）は現代の手書き日本語ならCER1〜3%と高精度。ただし縦書き活字はGPT-4oが「異様に低くなる」との報告があり、古文書くずし字は専用モデルでも不完全。横書き手書きはGoogle Document AIが優秀との定性評価もある。
- OCR後の構造化（人物・続柄・生年月日・婚姻関係の抽出）はLLMが得意領域で、JSON構造化出力との相性が良い。「OCRで座標＋テキスト抽出→LLMで意味構造化」の2段構成が定石。

### 3. 法規制・プライバシー
- 個人情報保護法上、戸籍は本籍地・出自など「本人に対する不当な差別・偏見が生じ得る情報」を含み得るため要配慮個人情報に準じた慎重な扱いが求められる。要配慮個人情報は取得時の本人同意が必須、オプトアウトによる第三者提供が禁止、漏えい時は個人情報保護委員会への報告・本人通知が義務。
- 戸籍法上、戸籍謄本の取得は本人・配偶者・直系親族等に限定され、戸籍法10条2項で「不当な目的」の場合は交付を拒める。2024年3月1日から広域交付制度で本人・配偶者・直系親族分は最寄り窓口でまとめて取得可能に。サービス側が取得代行すると戸籍法・職務上請求の問題に触れるため、利用者が自ら取得したスキャン画像を扱う設計にすべき。
- 家系図作成は最高裁第一小法廷 平成22年12月20日判決（刑集64巻8号1291頁、宮川光治裁判長）で「家系図作成について、行政書士の資格を有しない者が行うと国民生活や親族関係に混乱を生ずる危険があるという判断は大仰にすぎ、これを行政書士職の独占業務であるとすることは相当でない」と判示し、観賞用・記念用家系図は行政書士の独占業務ではないと確定。ただし相続関係説明図など「事実証明文書」は行政書士業務に該当し得るため、サービスは「観賞用・記念用」に位置づけるべき。
- 死者情報は個人情報保護法の対象外（第2条1項は「生存する個人に関する情報」に限定）。ただし戸籍は故人と生存する遺族の情報が混在し、生存者を識別できる情報（相続財産情報等）は「個人情報」として保護対象（個人情報保護委員会FAQ、最判平成31年3月18日）。
- 個人開発者が有償提供する場合、個人情報取扱事業者としてプライバシーポリシー、利用目的明示、安全管理措置、委託先（OCR/LLM API）の監督義務が発生。特に画像を外国のAPI（米国等）に送る場合、越境移転の同意取得が必要。

### 4. 技術スタック
- 可視化：**family-chart（D3.js製、MITライセンス、無料）**が家系図特化でおすすめ。React/Vue/Svelte/vanillaに対応。GoJS（商用ライセンス有償、genogram対応で養子・複数配偶者に強い、日本の皇室家系図デモあり）、dTree（多親対応・軽量）、Cytoscape.js、React Flowも選択肢。
- データ標準：GEDCOM（最新7.0、事実上の業界標準は5.5.1）対応でエクスポート/インポートに対応するとベンダーロックイン回避になり差別化ポイント。GEDCOM 7.0はFAM記録で養子・里親・事実婚等にも対応するよう仕様が拡張されている。TypeScript製のGEDCOMツールキットもnpmに存在。
- 無料版：IndexedDB（大きめデータ）/localStorageでクライアント完結、GEDCOM/JSONファイルのエクスポート・インポートでデータ永続化。サーバ保存なしを担保。静的ホスティング（Vercel/Netlify）。
- 有償版バックエンド：**Supabase**（PostgreSQL、無料枠MAU5万・DB500MB・ストレージ1GB、Pro月額25ドルの定額制で予算予測しやすい、RLSで行レベルアクセス制御、S3互換ストレージ）が家系図のリレーショナルデータ（人物・続柄・婚姻）と相性が良い。Firebaseはモバイル・リアルタイム向きだが従量課金でトラフィック急増時にコスト予測が難しい。開発者のReact Native経験を踏まえるとどちらも扱えるが、リレーショナル構造とコスト予測性からSupabase推奨。
- 共有：署名付きURL＋パスワード方式が現実的。E2E暗号化はパスワードから鍵導出しクライアント暗号化する方式が理想だが、パスワード紛失＝復旧不可というUXトレードオフがある。
- 決済：Stripeでサブスク/都度課金。freemiumは無料枠（クライアント完結）＋有償枠（戸籍処理・クラウド保存・共有）の明確な線引き。
- パイプライン：スキャン画像アップロード→前処理（傾き補正・地紋除去・二値化：OpenCV等）→OCR（Google Vision/Document AI）→LLM構造化（Gemini/GPT）→family-chartで家系図生成。

### 差別化ポイントの整理
ユーザーの不満から導かれる差別化軸：
1. **オンボーディング/チュートリアル徹底**（「使い方のイントロダクションがない」への対応）。
2. **自動整列＋崩れないレイアウト**（家系図特有の養子・再婚・複数配偶者に対応するアルゴリズム）。family-chart/GoJS採用。
3. **和暦⇄西暦の自動変換、旧字体対応の入力補助**（「西暦入力が面倒」への対応）。
4. **プライバシーファースト**（無料版はサーバ保存なし＝「個人情報をサーバーに送りたくない」層に刺さる。名字由来netの「公開・遠縁検索前提」への明確なアンチテーゼ）。
5. **高品質な印刷/PDF/巻物レイアウト出力**（親戚配布・贈答需要）。
6. **戸籍スキャンからの自動生成**（有償の目玉。既存の個人向けWebサービスにはほぼ存在しない体験。競合はB2B/士業向けのミツローくん等に限られる）。
7. **GEDCOMインポート/エクスポートでロックイン回避**（データ消失不安への対応）。

### OCR+LLM構造化のコスト概算（1件あたり）
戸籍1件＝仮に10ページ（除籍・改製原戸籍含む）と想定。
- Google Cloud Vision API（DOCUMENT_TEXT_DETECTION）：機能ごと1,000ユニット/月まで無料、以降1,000枚あたり約1.5ドル → 10ページ＝約0.015ドル（約2.3円）。
- または Google Document AI（Enterprise Document OCR）：1,000ページあたり約30ドル → 10ページ＝約0.30ドル（約45円）。
- LLM構造化（Gemini 2.5 Flash想定、入力$0.15/1Mトークン・出力$0.60/1Mトークン思考なし）：10ページOCR結果＋プロンプト＝仮に3万入力トークン＋5千出力トークン → 約0.0045ドル＋0.003ドル＝約0.0075ドル（約1.2円）。画像を直接LLMに入れる場合は1画像約1,290トークン。
- **合計：Vision＋Flash構成なら1件あたり数円〜十数円程度、Document AI採用でも1件50〜60円程度。** 十分に低コストで、有償価格（数百〜数千円）に対して原価率は極小。ただし精度検証・人間補正の工数が実質的な主コストになる。

### 推奨アーキテクチャ

**無料版（クライアント完結）**
- フロント：React（開発者のReact Native経験活用）+ family-chart（D3.js）。
- 状態/保存：IndexedDB＋localStorage。
- 入出力：GEDCOM/JSONエクスポート・インポート、PNG/PDF出力。
- サーバ通信なし（静的ホスティング）。

**有償版（BaaS）**
- 認証・DB・ストレージ：Supabase（PostgreSQLで人物・続柄をリレーショナルに管理、RLSでユーザー単位分離、S3互換ストレージに戸籍画像）。
- 処理：Supabase Edge Functions or Cloud Functionsから OCR API→LLM構造化を呼び出し。
- 共有：署名付きURL＋パスワード。可能ならクライアント側暗号化。
- 決済：Stripe（サブスク＋従量/都度課金）。

### 法的リスクと取るべき対応
- **戸籍取得には関与しない**：利用者が広域交付等で自ら取得した戸籍のスキャン画像のみを処理対象とし、取得代行・職務上請求には一切踏み込まない。
- **観賞用・記念用と明示**：相続関係説明図等の「事実証明文書」作成を業として請け負う体裁を避け、規約で用途を観賞用・家族記録用に限定。相続手続きへの利用は自己責任と明記。
- **個人情報取扱事業者としての整備**：プライバシーポリシー、利用目的明示、安全管理措置（暗号化・アクセス制御）、委託先（OCR/LLM事業者）の監督、越境移転の同意取得、漏えい時の報告体制。
- **要配慮情報の配慮**：本籍地・出自等に関する情報の取扱いに同意を得る。差別につながり得る情報の表示・共有に注意。
- **死者情報と生存者情報の混在**：故人情報は法対象外だが、同一戸籍の生存遺族は個人情報として保護対象。共有機能で第三者に渡る際の同意設計を組む。

## Recommendations

**段階1（MVP・0〜3ヶ月）：無料クライアント完結版でUX仮説を検証する。**
- family-chart＋IndexedDB＋GEDCOM入出力で、「崩れない自動整列」「和暦入力」「養子・再婚表現」「チュートリアル」を実装。
- 検証仮説：既存サービスの不満（操作性・レイアウト・プライバシー）を解消すれば無料ユーザーが定着するか。KPI：チュートリアル完了率、7日後の家系図保存継続率、人物登録数中央値。
- 閾値：週次アクティブ維持率が20%を超え、SNS等でオーガニック言及が出るなら次段階へ。

**段階2（3〜6ヶ月）：有償の戸籍自動生成を「平成6年式（活字・横書き）限定」で検証する。**
- Vision/Document AI＋Gemini/GPTで平成戸籍のみ自動構造化。古い手書き縦書き戸籍は「手入力補助＋部分OCR」のハイブリッドで割り切る（サムポローニアも縦書き手書きは手入力補完である点と同じ現実的判断）。
- 検証仮説：戸籍アップロード→自動生成体験に数百〜数千円を払う人がいるか。まず少額課金（1件買い切り or 月額）でStripe実装。
- 閾値：無料ユーザーの有償転換率、戸籍処理の自動抽出精度（人間補正なしで許容できる項目正答率）を測定。手書き古戸籍への拡張はTOPPAN等の動向とLLM精度向上を見て判断。

**段階3（6ヶ月〜）：クラウド保存＋パスワード共有を本実装し、法務・セキュリティを固める。**
- プライバシーポリシー・利用規約を専門家レビュー。安全管理措置文書化。越境移転同意フローを組込み。
- 共有はまず署名付きURL＋パスワード。E2E暗号化は需要次第で。

**判断を変える指標**：①無料版の定着が弱ければ有償に進まずUX改善に集中。②平成戸籍の自動抽出精度が実用に達しなければ「戸籍読み取り補助ツール」に位置づけを下げる。③手書き古戸籍の需要が強ければ、TOPPAN/サムポローニア級の技術が民生化するまで人力解読との協業（外注）で埋める。

## Caveats
- 戸籍そのもののOCR実測精度（CER等）の公開データは存在せず、企業発表値（サムポローニアの「60分→10分」等）はPR・自社/自治体発表で第三者検証がない。手書き古戸籍の自動化は2026年時点で各社「実用化を目指す」段階であり、個人開発での完全自動化は過度に楽観視すべきでない。
- API料金・BaaS料金・LLM料金は変動が激しく、本レポートの数値は調査時点の公開情報に基づく概算。実装前に各公式ページで再確認が必要。
- 法的整理は一般的な調査に基づくもので、有償サービス開始前に弁護士・行政書士等の専門家確認を強く推奨する。特に「観賞用・記念用」の線引きと相続実務利用の免責は要精査。
- 競合のユーザー不満はレビュー・ブログからの抽出であり、母数や代表性は限定的。MVP前に定量的なユーザーインタビュー/アンケートで裏付けるべき。
- 市場規模は終活関連（矢野経済研究所：2024年度の身元保証＋生前整理2分野合計で234.5億円、2025年度は前年比約110%の257.3億円見込み）や相続手続き需要（法定相続情報証明制度は2017年5月開始）から間接的に推測したもので、家系図作成サービス単体の市場規模統計は乏しい。2025年問題で年間死亡数が増加し大相続時代を迎えるため、追い風の環境ではある。
