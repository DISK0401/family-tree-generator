# Design: add-gedcom-json-import-export

## Context

リポジトリは Vite + React + TypeScript の雛形(プレースホルダーページ)のみで、家系図データを扱うコードは存在しない。本変更で内部データモデルと GEDCOM(7.0 / 5.5.1互換)・JSON の入出力を実装し、後続の編集UI・描画(family-chart)・IndexedDB永続化の土台を作る。

前提・制約:

- 無料版の絶対制約として、処理はすべてブラウザ内で完結させる(後述のデータフロー参照)。
- 内部データモデルは有償版(戸籍OCRの構造化出力・クラウド保存)でも共通利用する前提で、GEDCOMより柔軟な表現力を持たせる。
- 和暦・不完全日付・旧字体・複雑な家族関係(養子・再婚・複数配偶者)を表現できること。

### データフロー(必須明記事項)

- **すべての処理はクライアント(ブラウザ)内で完結する。** ファイルの読み込みは File API、書き出しは Blob + ダウンロードリンクで行い、家系図データ・個人情報はサーバ・外部APIへ一切送信しない。
- 本変更にネットワーク通信を伴うコードは存在しない(静的アセットの配信のみ)。Supabase・Gemini 等の外部サービスは不使用のため、RLS設計・外部API送信データの最小化方針は本変更では該当なし。
- データの存在場所: インポート元ファイル(ユーザー端末)→ ブラウザのメモリ上(内部データモデル)→ エクスポートファイル(ユーザー端末)。永続化(IndexedDB)は後続チェンジ。

## Goals / Non-Goals

**Goals:**

- 内部データモデル(TypeScript型 + 実行時バリデーション)の確立
- GEDCOM 7.0 / 5.5.1 のインポート、GEDCOM 7.0 / 5.5.1互換モードのエクスポート
- ネイティブJSON形式(スキーマバージョン付き)でのロスレスな入出力
- 非対応タグ・解釈不能データの保全と、インポート結果サマリ(警告一覧)の提示
- 日本語環境で流通するファイルの文字コード対応(文字化け防止)
- 最小限のファイル入出力UI(ファイル選択・ドラッグ&ドロップ・ダウンロード)

**Non-Goals:**

- IndexedDB永続化、家系図描画・編集UI、PNG/PDF出力(後続チェンジ)
- GEDCOMのANSELエンコーディング対応(日本語ファイルでの流通が稀なため。検出時は警告を出して読み込み中断)
- GEDCOMのマルチメディア(OBJE)ファイル本体の取り込み(参照情報のみ保全)
- 有償版機能(クラウド保存・戸籍OCR)との接続

## Decisions

### D1. GEDCOM処理は自前実装とする(ライブラリ不採用)

- **決定**: GEDCOM行構文(level / xref / tag / value + CONT/CONC)のパーサ・シリアライザを自前実装し、その上に内部データモデルとのマッピング層を置く。
- **理由**:
  - npmの既存ライブラリ(`read-gedcom`、`parse-gedcom` 等)は 5.5.1 の「パースのみ」が中心で、GEDCOM 7.0 対応・シリアライズ・非対応タグのラウンドトリップ保全という本変更の要件を満たすものがない。
  - GEDCOMの行構文自体は単純で、パーサコアの実装コストは小さい。仕様解釈の中心はマッピング層にあり、そこはライブラリを使っても自前で書く必要がある。
  - 依存を増やさないことで、ライセンス・保守停止リスクを回避する。
- **代替案**: `read-gedcom`(MIT, TS)を採用しインポートのみ委譲する案は、7.0非対応とエクスポート側の自前実装が結局必要になる点で却下。

### D2. 汎用GEDCOMツリーを中間表現とし、「構文層」と「意味層」を分離する

- **決定**: `GedcomNode { tag, value?, xref?, children[] }` の汎用ツリーを中間表現とし、①テキスト⇄ツリー(構文層)、②ツリー⇄内部データモデル(意味層)の2段構成にする。
- **理由**: 非対応タグの保全(未知のサブツリーをそのまま内部モデルに添付し再エクスポート時に復元)が自然に実現できる。7.0と5.5.1の差異(タグ名・構造の違い)は意味層のマッピングテーブルで吸収する。
- **代替案**: テキストから直接内部モデルへ変換する1段構成は、タグ保全とバージョン差異吸収が絡み合い複雑化するため却下。

### D3. 内部データモデルは「個人 + 家族(結合)」の2エンティティ構成

- **決定**: GEDCOMと同型の `Person`(個人)と `Family`(夫婦・親子の結合単位)を核とする。
  - `Person`: ID(UUID)、氏名(後述D5)、性別、生没(後述D4の日付型 + 場所)、イベント、メモ、GEDCOM非対応タグの保全領域。
  - `Family`: パートナー(0〜2名以上を許容する配列。複数配偶者は複数の`Family`レコードまたは同一レコード内配列で表現)、関係種別(婚姻・事実婚・離婚済み等)、子リンク。
  - 子リンクには **続柄種別**(実子 / 養子 / 継子 / 里子)を持たせ、養子縁組は実親`Family`と養親`Family`の両方への所属を許容する。再婚は複数の`Family`レコードで自然に表現される。
- **理由**: GEDCOM 7.0のINDI/FAM構造と相互変換しやすく、family-chart等の描画ライブラリのデータ形にも変換容易。要件の複雑な家族関係(養子・再婚・複数配偶者)を無理なく表現できる。
- **代替案**: 人物間の直接エッジ(親子・配偶者)だけを持つグラフモデルは柔軟だが、GEDCOMのFAM単位の情報(結婚日・離婚等)の置き場が不自然になるため却下。

### D4. 日付型は「構造化値 + 原文保持」のハイブリッド

- **決定**: `LifeDate { original: string, calendar: 'gregorian' | 'japanese', era?: string, year?, month?, day?, qualifier?: 'about' | 'before' | 'after' | 'estimated' }` のような構造とし、**入力された原文を必ず保持**する。
  - 和暦⇄西暦変換は自前の元号テーブル(明治以降は必須、江戸後期の主要元号〈文化・文政・天保〜慶応〉も収録)で行うユーティリティを実装する。
  - GEDCOMエクスポート時: 西暦変換可能なら標準のDATE値として出力し、和暦原文は 7.0 では `PHRASE`、5.5.1 では日付句(丸括弧表記)またはNOTEで保全する。
- **理由**: 戸籍由来のデータは和暦・不完全日付(「明治十年頃」等)が常態であり、原文を落とすと校正・信頼性が損なわれる。`Intl` の和暦サポートは整形専用でパースに使えないため、元号テーブルは自前で持つ。
- **代替案**: ISO 8601文字列のみで持つ案は不完全日付・和暦原文を表現できず却下。

### D5. 氏名型は「表記 + 読み仮名 + 別表記」の複合構造

- **決定**: `PersonName { family?: string, given?: string, kana?: { family?, given? }, romanized?, alternates?: PersonName[] }`。旧字体・異体字は**入力された文字をそのまま保持し、正規化(NFC変換や新字体への置換)を行わない**。旧字体/新字体の別表記は `alternates` に格納する。
  - GEDCOM 7.0 では `NAME` + `TRAN`(音訳。kana/romanized)にマッピング、5.5.1 では `ROMN` / 独自タグ `_KANA` 等の慣行タグを使用する。
- **理由**: 旧字体は家系記録として意味を持つため、暗黙の文字正規化はデータ破壊にあたる。読み仮名は日本語家系図の表示・並び替えに必須。
- **代替案**: 単一文字列 + フリガナのみの構造は、旧字体と新字体の併記需要(調査資料の競合不満点)を満たせず却下。

### D6. 文字コードの取り扱い

- **決定**:
  - **インポート**: BOMとGEDCOMヘッダの`CHAR`タグ、およびデコード試行(UTF-8厳格デコード→失敗時 Shift_JIS)で判定する。対応: UTF-8(BOM有無とも)、UTF-16(BOM必須)、Shift_JIS(国産ソフト由来ファイル対策。`TextDecoder('shift_jis')`使用)。ANSELは検出時に警告を出して中断する。
  - **エクスポート**: 常にUTF-8(BOM付き)。5.5.1互換モードでも `CHAR UTF-8` を出力する(5.5.1でも合法)。
- **理由**: 日本語GEDCOMの文字化けは移行体験を致命的に損なう。Shift_JISは規格外だが国産買い切りソフト由来のファイルで現実に流通しているため受け入れる。
- **代替案**: UTF-8のみ対応は移行ユースケースを狭めるため却下。

### D7. JSONネイティブ形式は内部モデルの直列化 + zodバリデーション

- **決定**: ネイティブJSONは `{ schemaVersion: 1, exportedAt, people: [...], families: [...], metadata }` とし、内部データモデルをそのまま直列化する。インポート時は `zod`(MIT)でスキーマ検証し、`schemaVersion` が未知の場合はエラー、既知の旧バージョンはマイグレーション関数で引き上げる(本変更ではv1のみ)。
- **理由**: JSONはGEDCOMで表現しきれない内部情報の完全バックアップが目的のため、内部モデルと1:1が最も単純で欠落がない。実行時検証は不正ファイル・改変ファイルからアプリ状態を守るために必須で、zodはTypeScript型との一体管理ができる。
- **代替案**: JSON Schema + ajv は型定義と二重管理になるため却下。検証なしの `JSON.parse` 直代入は安全性の観点で却下。

### D8. インポート結果サマリと警告の設計

- **決定**: インポートは `{ data, warnings: ImportWarning[] }` を返す。警告は「行番号・対象タグ・内容・処理(保全した/読み飛ばした)」を構造化して持ち、UIで人数サマリ(人物N名・家族M件)と警告一覧を平易な日本語で表示する。警告があってもインポート自体は成立させる(ベストエフォート方針)。エラーで中断するのは、GEDCOMとして解釈不能・非対応エンコーディング・JSONスキーマ不一致の場合のみ。
- **理由**: 競合(Gramps等)のエラーの難解さがUX不満点であり、「隠さず・怖がらせず」提示することが差別化になる。
- **代替案**: 厳格モード(警告=失敗)は他サービス産の不完全なファイルを実質受け入れられなくなるため却下。

### D9. エクスポートUIのデフォルトはGEDCOM 7.0、5.5.1互換を明示的に選択可能にする

- **決定**: エクスポート形式は「GEDCOM 7.0(推奨)/ GEDCOM 5.5.1互換(他サービス取込用)/ JSON(完全バックアップ)」の3択。UI上で各形式の用途を1行で説明する。
- **理由**: プロジェクト方針としては7.0が主軸だが、事実上の業界標準は5.5.1でありMyHeritage等への移行には5.5.1が安全。用途ラベルでユーザーが迷わず選べるようにする。
- **代替案**: デフォルト5.5.1は、7.0で拡張された養子・事実婚等の表現力を既定で捨てることになるため採らない。

### D10. 実行モデル: メインスレッドで同期処理(Web Workerは導入しない)

- **決定**: パース・シリアライズはメインスレッドで同期実行する。個人利用の家系図(数百〜数千人規模、数MB以下)では体感遅延が問題にならない想定。ファイルサイズ上限(例: 20MB)を設け、超過時は明示的にエラーとする。
- **理由**: Web Worker導入はビルド・テスト構成を複雑にする。ボトルネックが実測されてから後続チェンジで移行すればよい(パーサを純関数として実装しておけば移行コストは小さい)。
- **代替案**: 最初からWorker化する案は時期尚早な最適化として却下。

### モジュール構成(実装の指針)

```
src/
  domain/            # 内部データモデル(型・zodスキーマ・ID生成)
    person.ts family.ts date.ts name.ts
  lib/
    wareki/          # 元号テーブルと和暦⇄西暦変換
    gedcom/          # 構文層(parser/serializer)+ 意味層(mapper 7.0/5.5.1)
    json/            # ネイティブJSONの入出力・スキーマ検証・マイグレーション
  features/
    import-export/   # UIコンポーネント(ファイル選択・D&D・形式選択・結果サマリ)
```

## Risks / Trade-offs

- **[GEDCOM実装の仕様解釈誤り]** 自前実装のため、他サービスとの相互運用で想定外の差異が出る可能性 → 意味層をテーブル駆動にして修正を局所化。MyHeritage・Gramps・FamilySearch が出力したサンプルファイルでの相互運用テストをタスクに含める。
- **[非対応タグ保全の限界]** 5.5.1で読み込んだ独自タグを7.0で書き出す等、**バージョンをまたぐ再エクスポートでは保全情報の完全性を保証できない** → 同一バージョンでのラウンドトリップ保全を仕様上の保証範囲とし、バージョン変換時は変換されないタグを警告で明示する。
- **[Shift_JIS判定の誤検出]** エンコーディング自動判定は原理的に誤り得る → UTF-8厳格デコードを最優先し、失敗時のみShift_JISへフォールバック。判定結果をインポートサマリに表示し、文字化け時にユーザーが気づけるようにする。
- **[元号テーブルの網羅範囲]** 江戸期以前の元号・南北朝の並立元号までは収録しない → 変換不能な和暦は原文のまま保持し「変換できなかった」警告を出す(データは失わない)。テーブルはデータファイルとして追加容易にしておく。
- **[大きなファイルでのUIブロック]** 同期処理のため巨大ファイルで一時的にUIが固まる → サイズ上限で明示エラー。実測で問題になればWorker化(D10)。
- **[JSONスキーマの将来変更]** 内部モデル直列化のため、モデル変更がファイル互換に直結する → `schemaVersion` とマイグレーション関数の枠組みを初版から用意し、後方互換の読み込みを維持する。

## Migration Plan

- 新規機能のみで既存データ・既存ユーザーへの影響なし(永続化ストア未導入のため移行作業も不要)。
- 通常フロー(feature → develop でdev環境確認 → main で本番)でデプロイする。
- ロールバックは `git revert` + 再デプロイのみ。サーバ側リソースの変更はない。

## Open Questions

- 5.5.1互換モードでの読み仮名タグは `_KANA`(独自タグ慣行)と `ROMN` のどちらを主とするか。実装時にMyHeritage・Gramps の実ファイルを確認して決定する(どちらでも読み込めるようにはする)。
- エクスポートファイルに「親族の個人情報を含む」旨の注意書きをUIで表示する文言・タイミング(specsのシナリオ確定時に決定)。
