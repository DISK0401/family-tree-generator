# family-data-model

家系図の内部データモデル。人物・家族関係・日付・氏名の表現を定義し、GEDCOM/JSON入出力および後続の編集UI・描画・永続化の共通基盤となる。

## ADDED Requirements

### Requirement: 人物の表現
システムは人物(Person)を一意なID(UUID)を持つエンティティとして表現しなければならない(SHALL)。人物は氏名・性別・生没情報(日付と場所)・メモを保持でき、すべての属性は未入力(不明)を許容しなければならない(SHALL)。

#### Scenario: 属性が不明な人物の登録
- **WHEN** 氏名のみが判明している人物(生没年・性別不明)をデータモデルに登録する
- **THEN** バリデーションエラーにならず、不明な属性は「未設定」として保持される

#### Scenario: 人物IDの一意性
- **WHEN** 複数の人物を登録する
- **THEN** 各人物には衝突しない一意なIDが自動採番され、家族関係からIDで参照できる

### Requirement: 氏名の表現(旧字体・読み仮名)
氏名は「姓・名の表記」「読み仮名(かな)」「ローマ字表記」「別表記の配列」を持つ複合構造でなければならない(SHALL)。システムは入力された文字をそのまま保持し、旧字体・異体字を新字体へ変換する等の暗黙の文字正規化を行ってはならない(MUST NOT)。

#### Scenario: 旧字体の氏名の保持
- **WHEN** 姓「齋藤」(旧字体)を入力して保存し、再度読み出す
- **THEN** 「斎藤」「斉藤」等へ変換されず、入力どおり「齋藤」が返される

#### Scenario: 旧字体と新字体の併記
- **GIVEN** 姓の表記が「渡邊」の人物
- **WHEN** 別表記として「渡辺」を追加する
- **THEN** 主表記「渡邊」と別表記「渡辺」の両方が保持され、区別して取得できる

#### Scenario: 読み仮名の保持
- **WHEN** 姓「東海林」に読み仮名「しょうじ」を設定する
- **THEN** 表記と読み仮名が独立に保持され、読み仮名で取得できる

### Requirement: 家族関係の表現
システムは家族(Family)を「パートナーの集合と子リンクの集合を持つ結合単位」として表現しなければならない(SHALL)。家族は関係種別(婚姻・事実婚・離婚成立済み等)と、結婚日・離婚日等のイベントを保持できなければならない(SHALL)。一人の人物は複数の家族にパートナーとして所属できなければならない(SHALL)。

#### Scenario: 再婚の表現
- **GIVEN** 人物Aが人物Bと離婚後、人物Cと再婚している
- **WHEN** A-Bの家族(離婚成立済み)とA-Cの家族(婚姻中)を登録する
- **THEN** Aは2つの家族にパートナーとして所属し、それぞれの関係種別・イベントが独立に保持される

#### Scenario: 事実婚の表現
- **WHEN** 婚姻届を出していないパートナー関係を関係種別「事実婚」で登録する
- **THEN** 婚姻と区別された関係種別として保持される

#### Scenario: 片親のみ判明している家族
- **WHEN** 母親のみ判明している親子関係を、パートナーが1名の家族として登録する
- **THEN** エラーにならず、子から母親のみが親として辿れる

### Requirement: 続柄種別と養子縁組の表現
家族への子リンクは続柄種別(実子・養子・継子・里子)を持たなければならない(SHALL)。一人の子は複数の家族に異なる続柄種別で所属できなければならない(SHALL)(例: 実親の家族に「実子」、養親の家族に「養子」)。

#### Scenario: 養子縁組の二重所属
- **GIVEN** 人物Xが実父母の家族F1に実子として属し、養父母の家族F2と養子縁組している
- **WHEN** XをF1に続柄「実子」、F2に続柄「養子」で登録する
- **THEN** Xから実親と養親の両方が続柄種別付きで辿れ、どちらの関係も失われない

#### Scenario: 継子の表現
- **GIVEN** 人物Aと再婚した人物Cに、Aの前婚の子Yがいる
- **WHEN** YをA-Cの家族に続柄「継子」で登録する
- **THEN** YはA-Cの家族から続柄「継子」として辿れ、実親の家族の続柄「実子」と区別される

### Requirement: 日付の表現(不完全日付・原文保持)
日付は「入力原文」「暦種別(西暦・和暦)」「元号・年・月・日(いずれも省略可)」「修飾子(頃・以前・以後・推定)」を持つ構造でなければならない(SHALL)。システムは入力原文を必ず保持し、構造化に失敗した場合でも原文を失ってはならない(MUST NOT)。

#### Scenario: 和暦の不完全日付
- **WHEN** 生年月日として「明治十年頃」を登録する
- **THEN** 元号「明治」・年「10」・修飾子「頃」が構造化され、原文「明治十年頃」もそのまま保持される

#### Scenario: 年のみの日付
- **WHEN** 没年として「1923」(年のみ)を登録する
- **THEN** 月日が未設定の有効な日付として保持される

#### Scenario: 構造化できない日付原文の保全
- **WHEN** 「戌年生まれ」のように構造化できない日付表現を登録する
- **THEN** エラーで失われず、原文のみの日付として保持される

### Requirement: 和暦⇄西暦の相互変換
システムは元号テーブル(明治以降の全元号、および文化〜慶応の江戸後期主要元号)に基づき、和暦と西暦を相互変換するユーティリティを提供しなければならない(SHALL)。変換できない元号・日付の場合、変換失敗を明示的に返し、元データを変更してはならない(MUST NOT)。

#### Scenario: 和暦から西暦への変換
- **WHEN** 「昭和64年1月7日」を西暦に変換する
- **THEN** 「1989年1月7日」が返される

#### Scenario: 改元年の西暦から和暦への変換
- **WHEN** 「1989年1月8日」を和暦に変換する
- **THEN** 「平成元年1月8日」が返される(1月7日以前は昭和64年と変換される)

#### Scenario: テーブルにない元号の変換失敗
- **WHEN** 収録範囲外の元号「應永3年」を西暦に変換しようとする
- **THEN** 変換失敗が明示的に返され、呼び出し元で原文のまま扱える

### Requirement: 参照整合性の検証
システムはデータモデル全体(人物・家族の集合)に対する実行時バリデーションを提供しなければならない(SHALL)。家族が参照する人物IDが存在しない場合、バリデーションは整合性エラーを検出しなければならない(SHALL)。

#### Scenario: 存在しない人物への参照の検出
- **GIVEN** 家族F1が子として人物ID「p-999」を参照しているが、該当人物が存在しない
- **WHEN** データモデル全体をバリデーションする
- **THEN** F1の参照先欠落が整合性エラーとして報告される

#### Scenario: 正常なデータモデルの検証
- **WHEN** すべての参照が解決できるデータモデルをバリデーションする
- **THEN** エラーなしで成功する
