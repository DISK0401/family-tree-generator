# json-import-export

アプリネイティブJSON形式(スキーマバージョン付き)によるロスレスなエクスポート/インポート。GEDCOMで表現しきれない内部情報の完全バックアップ手段。

## ADDED Requirements

### Requirement: クライアント完結の処理(プライバシー)
JSONのインポート・エクスポート処理は、すべてユーザーのブラウザ内で完結しなければならない(MUST)。処理の過程で家系図データ・ファイル内容を含むいかなる情報もサーバ・外部APIへ送信してはならない(MUST NOT)。

#### Scenario: インポート/エクスポート時にネットワーク送信が発生しない
- **WHEN** JSONファイルのインポートおよびエクスポートを実行し、ブラウザ開発者ツールのNetworkタブを確認する
- **THEN** 自ホストの静的アセット取得以外のネットワーク通信が発生しない

### Requirement: ネイティブJSONエクスポート
システムは内部データモデルをネイティブJSON形式でエクスポートできなければならない(SHALL)。出力は `schemaVersion`(整数)・エクスポート日時・人物配列・家族配列を含み、内部データモデルの全情報(保全済みGEDCOMタグ・日付原文・別表記を含む)をロスレスに直列化しなければならない(SHALL)。

#### Scenario: 全情報を含むエクスポート
- **GIVEN** 旧字体の別表記・和暦原文の日付・GEDCOM由来の保全タグを含むデータモデル
- **WHEN** JSON形式でエクスポートする
- **THEN** schemaVersionを含むJSONファイルがダウンロードされ、別表記・日付原文・保全タグがすべて含まれる

### Requirement: ネイティブJSONインポートとスキーマ検証
システムはネイティブJSONファイルをインポートし、スキーマ検証に合格した場合のみ内部データモデルへ復元しなければならない(SHALL)。スキーマ不一致・`schemaVersion` 欠落・未知の `schemaVersion` の場合はインポートを中断し、理由を平易な日本語で表示し、既存データを変更してはならない(MUST NOT)。

#### Scenario: 正常なJSONのインポート
- **WHEN** 本システムがエクスポートしたJSONファイルをインポートする
- **THEN** スキーマ検証に合格し、人物・家族が内部データモデルへ復元される

#### Scenario: スキーマ不一致の拒否
- **WHEN** 必須フィールドが欠けた・型が不正なJSONファイルをインポートしようとする
- **THEN** インポートは中断され、どの項目が不正かが平易な日本語で表示され、既存データは変更されない

#### Scenario: 未知のschemaVersionの拒否
- **WHEN** 現在の実装が知らない `schemaVersion`(例: 999)のJSONファイルをインポートしようとする
- **THEN** インポートは中断され、「新しいバージョンのアプリでエクスポートされた可能性がある」旨が表示される

### Requirement: スキーマバージョンとマイグレーションの枠組み
JSONインポートは、過去の既知の `schemaVersion` のファイルを現行スキーマへ引き上げるマイグレーションの枠組みを備えなければならない(SHALL)。初版時点では現行バージョン(1)のみを既知とする。

#### Scenario: 現行バージョンのそのまま受理
- **WHEN** `schemaVersion: 1` のJSONファイルをインポートする
- **THEN** マイグレーションなしでそのまま復元される

### Requirement: ラウンドトリップの完全性
本システムでエクスポートしたJSONを再インポートした場合、内部データモデルは元と等価に復元されなければならない(SHALL)。GEDCOM形式では警告付きで代替表現になる情報(事実婚・保全タグ・日付原文等)も、JSONでは欠落なく復元されなければならない(SHALL)。

#### Scenario: エクスポート→インポートの等価復元
- **GIVEN** 養子縁組・再婚・事実婚・和暦原文日付・旧字体別表記を含む複雑なデータモデル
- **WHEN** JSON形式でエクスポートし、そのファイルを再インポートする
- **THEN** 復元されたデータモデルは元のデータモデルと等価である(全人物・全家族・全属性が一致する)
