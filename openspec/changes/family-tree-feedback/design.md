## Context

本変更は `proposal.md` に列挙した無料版フィードバックへの対応であり、対象コードはすべて `src/` 配下のクライアントサイドUI・描画・ローカルファイル入出力と、`package.json`/Vite設定などの開発環境設定に閉じる。データフローの観点では、本変更によって新たにサーバ・外部APIへ送信される情報は一切ない(既存の「クライアント完結」を維持する UI/表示ロジック/ファイル名生成/開発環境設定の変更のみ)。Supabase・Gemini等の外部サービスを利用する機能は本変更に含まれないため、RLSポリシー設計・外部API送信データの最小化方針は適用対象外(N/A)。

調査により、フィードバックの多くが同一の技術的根本原因を共有していることが分かった:

1. **`main_id` 追従による再描画**(`src/rendering/FamilyTreeCanvas.tsx` L117-125): 人物カードをクリックするたびに `chart.updateMainId(findRootAncestor(...))` を呼び、選択人物の最上位祖先を新たな `main_id` としてツリー全体を再構築している。family-chart は単一視点(`main_id` を根とする祖先/子孫展開)の描画方式のため、`main_id` が変わるたびにツリーの走査順序が変わり、**子・配偶者の左右並び順が視点ごとに変化しうる**(操作感1のバグ)。
2. **家族の重複所属時の「主たる親子線」選択**(`src/rendering/to-family-chart-data.ts` L93-102): 同一人物が複数の `Family` に子として属する場合(実親+養親など)、`Object.values(doc.families)` の出現順で最初に見つかった家族のみを親子線として採用する。夏目漱石サンプル(`src/samples/data/natsume-soseki.ts`)では実親家族 `f-natsume` が養親家族 `f-shiobara` より先に定義されているため、**養子縁組の系線が描画されず実親側のみが表示される**(バグ1)。この問題は夏目漱石固有ではなく、「実親情報も記録されている養子」全般で発生する構造的な問題。
3. **フォームのローカル状態 + `key={personId}` remount パターン**(`src/components/PersonPanel.tsx` L134, `src/components/PersonEditForm.tsx`): 人物編集フォームは `useState` によるアンコントロールド状態を持ち、`<PersonEditForm key={personId} .../>` としているため、選択人物が変わると React がフォームを丸ごとアンマウント/リマウントする。これにより**未確定の入力が警告なく静かに破棄される**(操作感3のバグの真因)。
4. **エクスポートファイル名の固定文字列**(`src/components/ImportExportControl.tsx` L191-201): `family-tree.json` / `family-tree.ged` のように固定名で `downloadBytes`/`downloadText`(`src/features/import-export/fileIO.ts`)を呼んでいるため、複数回エクスポートすると同名ファイルが上書きされる。
5. **`@rolldown/binding-*` の Windows 向けロックファイルエントリ欠落**: `package-lock.json` には `@rolldown/binding-linux-x64-gnu`/`-linux-x64-musl` の具象パッケージエントリのみが存在し(ロックファイル生成環境がLinuxだったため)、`win32-x64-msvc` 等の他プラットフォーム分が欠落している。npmの `optionalDependencies` は、ロックファイル生成時のプラットフォームに存在するものだけが `node_modules/...` エントリとして書き込まれる既知の挙動があり、Windows環境で `npm install` してもネイティブバイナリが解決されない(rollup/rolldown系パッケージで頻出する既知の npm の挙動)。
6. **ブランディングの未反映**: `index.html` の `<title>`・OGPは正しく「家系図帖」だが、`src/App.tsx` L48 のエディタヘッダーは `<h1 className="app-title">家系図</h1>` とだけ表示しており製品名が反映されていない。また `.app-header`(`src/App.css`)と `.landing-header`(`src/pages/LandingPage.css`)は `padding` に異なるデザイントークン(`--space-2/--space-4` 対 `--space-3/--space-5`)を使っており、見た目の高さが揃っていない。ファビコン(`public/favicon.svg`)は抽象的な紫の図形で、家系図帖のブランドを想起させない汎用的な意匠になっている。

## Goals / Non-Goals

**Goals:**
- `proposal.md` の全項目(操作感4・デザイン5・ファイル書き出し1・ブランディング3・バグ3)を、既存の家系図データモデル(`TreeDocument`)・保存形式(IndexedDB/GEDCOM/JSON)を変更せずに実現する
- 上記1〜3の構造的原因に対して、個別のパッチではなく再発防止になる修正を行う(特に子/配偶者の並び順は「どのノードが `main_id` か」に依存しない決定的なロジックにする)
- 新規 `display-settings` 機能を、既存の `TreeDocument` 永続化(IndexedDB)とは独立したローカル専用のUI設定として実装する

**Non-Goals:**
- family-chart 以外の描画ライブラリへの置き換え、または家系図キャンバスの全面リライト
- 同一ドキュメント内に複数の完全に非連結な家系(共通の祖先・配偶者関係を持たない複数の家族クラスタ)を単一画面へ同時最適配置する汎用レイアウトの実現(全体表示モードのスコープ限定。後述)
- 有償版(戸籍OCR・クラウド保存・共有)機能への変更
- GEDCOM/JSONのデータスキーマ変更(ファイル名生成のみが対象)

## Decisions

### D1. 子・配偶者の並び順は family-chart 公式のソートフックで決定する
**(スパイク調査により当初案から変更。詳細はOpen Questions下部の調査結果を参照)** `toFamilyChartData`(`to-family-chart-data.ts`)が生成する配列を事前ソートするだけでは不十分と判明した。family-chart は `calculateTree` 内で、利用者が渡した `sortChildrenFunction` を適用した**あと**に、必ず自前の `sortChildrenWithSpouses`(親の `gender === 'M'` かどうかで昇順/降順が反転する、配偶者インデックスによる再グルーピング)を実行する。この関数こそが「性別未設定の人物をクリックすると並び順が入れ替わる」バグの直接の原因である(半きょうだいが混在する家族で、`gender` が `'M'` でない親(女性・不明)のカードが描画起点になったときだけ順序が反転する)。

そのため、並び順は `FamilyTreeCanvas.tsx` で `chart.setSortChildrenFunction(...)` / `chart.setSortSpousesFunction(...)` という公式フックに実装する(アダプタ側で配列を並べ替える処理は行わない)。
- 子の並び順関数: `(生年月日の year 昇順) → (未設定の場合は displayName の辞書順)` の複合キーで比較する
- 配偶者の並び順関数: 婚姻イベント日付(`events` 内 `type: 'marriage'` の `date.date.year`)昇順。日付不明な婚姻は登録順(`data_stash` 内の出現順)を維持する
- 単婚(半きょうだいがいない)の子リストでは `sortChildrenWithSpouses` の比較キーが全員同値になり、V8の安定ソートにより上記フックの並び順がそのまま維持される。半きょうだいが混在するケース(複数婚)では `sortChildrenWithSpouses` が婚姻(配偶者)単位でグルーピングし直すため、完全な生年横断ソートにはならない点は既知の制約として許容する(通常の系図表現でも「どの配偶者との子か」でまとまって表示される方が読みやすいため、実害は小さいと判断する)

この並び順はソース(`TreeDocument`)のみから決定される比較関数として実装され、`main_id` やクリック操作に依存しないため、**操作感1のバグ(並び順の入れ替わり)とデザイン5(子の生年順表示)を同一の実装で解決する**。

**代替案として検討したもの**: family-chart 側のソートオプション設定(存在すれば)を使う案は、ライブラリが `main_id` 起点の走査順をどこまでカスタマイズ可能か不明なため採用しない。アダプタ層(ドメインモデル→描画データの一方向変換、design.md D1 by 家系図作成UI change)で完結させる方が既存アーキテクチャと整合する。

### D2. 複数家族所属時の「主たる親子線」選択を、実子より非実子(養子等)を優先する
`parentsByChild`(`to-family-chart-data.ts` L98-101)の「最初に見つかった家族を採用する」ロジックを、`pedigree !== 'biological'`(養子・継子・里子・不明)を **優先** して採用するよう変更する。実子関係は「特筆すべき情報がない」デフォルトの関係である一方、養子等の関係は家系図上で明示的に伝えたい情報であるため、表示上の優先度を非実子側に置く。

- 同一人物が複数の非実子関係を持つ(通常想定しない)場合は、従来通り `Object.values(doc.families)` の出現順にフォールバックする
- 選ばれなかった側の家族関係は `TreeDocument` には引き続き保持され、失われない(データモデルは無傷)。表示上どちらか一方の線しか引けない制約は family-chart 起因の既存の限界であり、本変更ではこの制約自体は解消しない(Non-Goals)

**代替案として検討したもの**: 同一人物を複数箇所に「分身ノード」として複製描画する方式は、family-chart の `id` ベースのD3データ結合(`FamilyTreeCanvas.tsx` のコメント参照)と衝突しリスクが高いため見送る。優先順位ルールによる解決は実装コストが小さく、フィードバックの実害(「養子が表示されない」)を直接解消する。

**実装時に判明した追加修正(D1と同様の理由)**: `parentsByChild` は `rels.parents`(カードの「主たる親」ポインタ)にのみ作用し、`rels.children`(親→子の forward ポインタ)は依然として全家族分の子を保持する。そのため、実装当初は次の2つの副作用が残っていた。
1. **系線スタイルの誤り**: `markLinkStyles` が人物単位の `pedigree` 値だけで破線/実線を決めていたため、実親側の辺(本来は実線)まで人物カードの `pedigree`(養子縁組後は`adopted`)に引きずられて破線になってしまう。→ `buildPedigreeByEdge(doc)` で親子ペア単位の続柄マップを構築し、`markLinkStyles` を辺(source/targetの実際の親子ペア)単位の判定に変更した。
2. **`findRootAncestor` が非実子優先ロジックと無関係だった**: `findRootAncestor` は独自に「最初に見つかった家族」で祖先をたどっており、`parentsByChild` の優先順位変更と食い違っていた(クリックで祖先へ追従しても養親側にはたどり着かない)。→ 優先順位判定を `findPrimaryParentFamily(doc, childId)` として切り出し、`toFamilyChartData` の `parentsByChild` 相当のロジックと `findRootAncestor` の両方から同じ優先順位で参照するようにした。これにより、養子縁組を持つ人物(またはその子孫)をクリックすると、`main_id` が養親側の祖先へ追従し、養親が表示範囲に入り、実親側は非表示人数バッジ(D6)で示される。

### D3. 未確定変更の検知と確認ダイアログ
`PersonEditForm` に `onDirtyChange(isDirty: boolean)` コールバックを追加し、ローカル状態(氏名・性別・生没日等)が `person` プロパティと異なる場合に `true` を通知する。`PersonPanel` はこれを中継し、`App.tsx` が選択人物切り替え(`onSelectPerson`)・パネルを閉じる操作(`onClose`)を **直接 `setSelectedPersonId` を呼ばず**、次のガード関数を経由するよう変更する:

```
requestChangeSelection(next) {
  if (isDirty) { setPendingSelection(next); /* ダイアログ表示 */ }
  else { setSelectedPersonId(next) }
}
```

確認ダイアログは既存の `confirm-dialog` パターン(`App.tsx` のサンプル上書き確認ダイアログと同一の `.confirm-dialog-overlay`/`.confirm-dialog` CSS)を再利用し、以下の3アクションを提供する: 「編集を続ける(キャンセル)」「変更を破棄して移動」「保存して移動」。「保存して移動」は `PersonEditForm` の保存処理(`onSave`)を先に実行してから選択を切り替える。

キーボード操作: ダイアログの主要アクションボタン(「保存して移動」)に `autoFocus` を設定し、ネイティブの「フォーカス中のボタンは Enter で押せる」挙動を利用する(カスタムの keydown ハンドラは追加しない)。`PersonEditForm` 自体は既に `<form>` + `<button type="submit">確定</button>` の構造のため、単一行テキスト入力にフォーカスがある状態での Enter は標準のフォーム送信仕様により既に「確定」を実行する。実装時に、`WarekiDateInput` 等のテキスト入力がこの標準動作を妨げていないことを回帰確認する(意図的な `preventDefault` は現状コード上見当たらない)。

### D4. 日付入力: 8桁数字形式のみ新規対応、ハイフン区切りは対応済み
`src/domain/parse-date.ts` の `GREGORIAN_SEP_RE`(`/^(\d{3,4})(?:[/-](\d{1,2})(?:[/-](\d{1,2}))?)?$/`)は既に `2000-01-01` 形式を受理しており、`parse-date.test.ts` にも回帰テストが存在する。実装が必要なのは「区切りなし8桁」(`20000101`)のみで、`GREGORIAN_SEP_RE` に隣接する新パターン `GREGORIAN_COMPACT_RE = /^(\d{4})(\d{2})(\d{2})$/` を追加し、`parseCore` 内で他のパターンにマッチしなかった場合に試す。4桁の西暦以外(2桁年など)は曖昧さを避けるため対象外とする。

### D5. 全体表示モード(折りたたみを抑止する表示)
**(スパイク調査により当初案から具体化。詳細はOpen Questions下部の調査結果を参照)** family-chart は `ancestry_depth`/`progeny_depth`(祖先・子孫の表示世代数)を **デフォルトで無制限** に描画する。祖先側・子孫側の深さ自体は元々制限されておらず、現行実装で傍系親族が漏れるのは、(a) `main_id` をクリックのたびに再計算・移動させているため視点ごとに見える範囲が変わって見えること、(b) family-chart は `main_id` 本人の兄弟姉妹をデフォルトでは描画しない(`show_siblings_of_main: false` がデフォルト)ことの2点が主因と判明した。

これを踏まえ、全体表示モードは以下の組み合わせで実装する(専用のツリー再構築ロジックは不要):
- `chart.setShowSiblingsOfMain(true)` を常時有効にし、`main_id` 本人の兄弟姉妹(と、`main_id` が祖先へ追従することで各世代の兄弟姉妹)も描画対象に含める
- 全体表示モードが有効な間、クリック時の `chart.updateMainId(...)` 呼び出しをスキップし、視点(=表示範囲)を固定したまま選択状態のみ更新する。固定する `main_id` は、モード有効化時点で「ドキュメント内で最大の連結範囲(祖先・子孫・配偶者を辿って到達できる人数が最大)を持つ根」を1回だけ算出する(このBFSはfamily-chart側にAPIがなく既存設計どおり自前実装が必要)

**既知の限界**: family-chart は単一 `main_id` を起点とする祖先/子孫+`main_id`本人の兄弟姉妹の走査でツリーを構築するため、`TreeDocument` 内に完全に非連結な複数の家族クラスタ(共通の血縁・婚姻関係を持たない別々の家系)が存在する場合、全体表示モードであっても最大クラスタ以外は表示されない(Non-Goals)。また「本人の兄弟姉妹」以外の傍系(祖父母の兄弟姉妹とその子孫等)は、`main_id` が祖先側へ追従する経路上でのみ間接的にカバーされ、完全な網羅は保証されない。通常の1家系図の利用シナリオでは実用上の影響は小さいと想定する。

**実装時に判明した追加修正(ユーザーフィードバック反映)**:
- `findMaxCoverageRoot`/`findRootAncestor` はいずれも「親を持たない最上位祖先」を返す設計のため、`show_siblings_of_main` が対象とする「main_id本人の兄弟姉妹」は、そのデータモデル上ほぼ常に0人になる(親が記録されていない人物には兄弟姉妹も記録され得ないため)。そのため全体表示モードの有効/無効による見た目の差が、非表示バッジの有無以外にほとんど生じないケースが多く、「切り替えても何も起きない」という体感につながっていた。この限界はD6のバッジクリック機能(下記)で実用上の不満点を解消する形で補っている
- 無効化(「折りたたみ表示に戻す」)時に `main_id` を明示的に戻していなかったため、次にカードをクリックするまで表示が変化せず、無効化操作自体が機能していないように見える不具合があった。選択中の人物がいる場合は、無効化と同時に `chart.updateMainId(findRootAncestor(doc, selectedPersonId))` を呼び、即座に折りたたみ表示相当の視点へ戻すよう修正した
- トグルボタン(`.tree-corner-panel` 内)と凡例(`.tree-legend`)が同じ絶対配置座標(`left`/`bottom`)を共有しており重なっていたため、両者を1つの縦積みフレックスコンテナにまとめ、重ならないよう修正した。あわせて凡例に性別インジケーター(D7)の凡例(男・女・不明の形状)を追加した

### D6. 折りたたみ時の非表示人数バッジ
デフォルト表示モードで、現在描画されている人物集合(`toFamilyChartData` の出力に含まれるID集合)を基準に、各人物の隣接関係(配偶者・親・子。`TreeDocument` 全体から算出)のうち描画集合に含まれないものを「境界」として検出する。境界ごとに、そこから到達可能かつ現在の描画集合に含まれない人物数を(すでにカウント済みの集合を除きながら)幅優先探索で数え、対応するカードに人数バッジ(例: 「+3」)を表示する。全体表示モードが有効な間はこのバッジは表示しない(原則として非表示人物が存在しないため)。

**実装時に判明した追加要件(ユーザーフィードバック反映)**: D2の「非実子を優先」ルールにより、養子縁組を持つ人物をクリックすると視点が養親側へ固定され、実親側は非表示バッジになる。しかし `findRootAncestor` の優先順位は決定的(常に非実子を優先)なため、一度養親側へ移動すると再クリックしても実親側には戻れない(逆方向へたどる手段がない)というデッドエンドが生じていた。これを解消するため、非表示人数バッジ自体をクリック可能にし、`computeHiddenCounts` が境界ごとに直接の非表示隣接人物ID(`revealId`)も返すようにした。バッジをクリックすると、選択状態は変えずに `chart.updateMainId(findRootAncestor(doc, revealId))` で視点だけをその人物の祖先側へ再追従させる。これにより「実親側 ⇄ 養親側」をバッジのクリックで行き来できるようになった。

### D7. 性別の視覚表現
現状、人物カードは性別による色分けを一切行っていない(`FamilyTreeCanvas.tsx` のコメント「選択状態のみを朱で表現する。性別による色分けはしない」という既存の意図的な設計判断による)。カードは墨(文字)・楮紙(背景)のみで、性別を示す視覚要素は一切ない。本変更ではこの方針を見直し、性別インジケーターを新規に追加する。色のみに依存すると色覚特性によっては判別できないため、アイコン・形状を主とし、色は補助的に使う(選択強調用の「朱」とは別の既存トークンを用い、朱との混同を避ける)。「不明」は性別未入力時のデフォルト値のままとし、無理に推測しない。

### D8. 現年齢・没年齢の表示
生年月日が判明している人物について、`birth.date.date` から現在日時(または `death.date.date`。没年月日が判明している場合)までの満年齢を計算し、カードに補助情報として表示する。生年月日が不明(`FuzzyDate` の `qualifier` が `about`/`before`/`after`/`between` などで年のみしか特定できない場合を含む)なケースは、計算可能な最も安全な下限・上限からの概算である旨が誤解を招かないよう、**年のみ判明している場合は年齢を表示しない**(D9の表示粒度設定で「年のみ」を選んでいても、年齢計算には日単位までの精度が必要なため独立に扱う)。

### D9. 表示設定(`display-settings`)の永続化
生年月日・没年月日のカード表示粒度(年のみ/年月まで/年月日まで)を含む表示設定は、家系図データ(`TreeDocument`)とは独立した端末ローカルのUI設定として `localStorage` に保存する(IndexedDBの `TreeDocument` 永続化層とは分離し、undo/redo履歴の対象にもしない)。GEDCOM/JSONのエクスポート内容には影響しない(エクスポートは常にドメインモデルの正確な値を出力し、表示設定は描画時にのみ適用する)。

### D10. エクスポートファイル名のタイムスタンプ付与
`ImportExportControl.tsx` のファイル名生成箇所を `family-tree-{timestamp}.json` / `family-tree-{timestamp}.ged` / `family-tree-{timestamp}-5.5.1.ged` に変更する。ユーザー指定の書式「yyyymmddhhss」は分(mm)を含まないため同一時間内の複数回エクスポートで衝突しうると判断し、実装では分を含む14桁 `yyyyMMddHHmmss`(例: `20260718143022`)を採用する。ローカル時刻(端末のタイムゾーン)を用いる。

### D11. Windows開発環境: ロックファイルへの全プラットフォーム分エントリの明示
ルート `package.json` に、`rolldown` が要求する主要プラットフォーム向け `@rolldown/binding-*` パッケージを `optionalDependencies` として明示的に追加する(現在解決されているバージョンに合わせてピン留め)。npmは**ルートの `package.json` に明示された** `optionalDependencies` については、ロックファイル生成環境のプラットフォームに関わらずロックファイルへエントリを書き込むため、Windows環境でも `npm install` のみでネイティブバイナリが解決されるようになる(この修正のためだけに `package-lock.json` を再生成する)。対象プラットフォームは開発機として現実的な `win32-x64-msvc` / `darwin-arm64` / `darwin-x64` / `linux-x64-gnu` を最小セットとする。

**代替案として検討したもの**: rolldown を使わない Vite の設定に切り替える案は、Vite 8 が内部でrolldownを標準採用しているため設定変更では回避できず見送る。`postinstall` スクリプトでの検出・自動インストールは、オフライン開発環境やCI環境で追加のネットワークアクセスを発生させる副作用があるため、より確実で追加処理の要らないロックファイル修正を優先する。

### D12. ブランディング統一
- `public/favicon.svg` を、既存デザイントークン(墨・楮紙・藍・朱)を用いた「家系図帖」を想起させる意匠(例: 縦書きの文字や系図の線を抽象化したモチーフ)へ差し替える。具体的なビジュアルは実装時に既存のデザイントークンと整合させて作成する
- `.app-header`(`App.css`)と `.landing-header`(`LandingPage.css`)の `padding` を共通のデザイントークン値に揃える(どちらの値に統一するかは実装時に見た目のバランスで決定するが、トークンを共有し将来の乖離を防ぐため両者が同じCSS変数を参照するようにする)
- `App.tsx` のエディタヘッダーの見出しを「家系図」から「家系図帖」に変更する

**実装時に判明した追加修正**: `padding` トークンを揃えただけでは、内部要素(ランディングのCTAボタンの縦padding・ボーダー分)の高さの違いにより、ヘッダー全体の高さが一致しなかった(実測: ランディング約65px・エディタ約55px)。`--header-min-height`(66px)を共通トークンとして新設し、両ヘッダーに `min-height` として適用することで、内部コンテンツの違いによらず高さが完全に一致するようにした。

## Risks / Trade-offs

- **[全体表示モードの限界]** family-chart の単一視点走査モデルにより、非連結な複数家系クラスタを含むドキュメントでは全体表示モードでも一部が表示されない可能性がある → D5のスパイクで技術的な実現可能性を先に確認し、無理な場合は「最大クラスタを表示し、他クラスタの存在をメッセージで明示する」形にフォールバックする
- **[D2の優先順位ルールが逆効果になるケース]** 「非実子を優先」ルールは大半のケースで意図通りだが、同一人物が2つの非実子関係(例: 養子縁組の解消後に別の家庭へ再度養子に出た等)を持つ場合、意図しない方が選ばれる可能性がある → 発生頻度は低いと判断し、既存の「出現順」フォールバックのまま許容する(既存の制約と同水準)
- **[D6の非表示人数バッジの計算コスト]** 人物数が多い家系図では境界ごとのBFSがO(N)を複数回走る可能性がある → 家系図は「高々数百人物」規模を前提とする既存設計(tree-store.ts のコメント参照)のため、実用上の性能問題にはならないと判断する
- **[D11のロックファイル再生成]** `package-lock.json` の全面再生成は差分が大きくなり、意図しない依存バージョンの変動を含むリスクがある → 再生成後は `npm run lint`/`npm run typecheck`/`npm test`/`npm run build` を通し、`git diff` で `optionalDependencies` 追加以外の予期しない変更(特にメジャーバージョン変動)が無いことを確認してからコミットする

## Migration Plan

データモデル・永続化スキーマの変更を伴わないため、特別なマイグレーション手順は不要。`display-settings` は新規のローカル設定のため、未設定時のデフォルト値(例: 年月日まで表示)を用意し、既存ユーザーの初回表示が変わらないようにする。

デプロイは既存の `cloudflare-deploy`(develop→dev環境、main→本番)フローに従う。問題発生時は各コミット(またはマージコミット)を `git revert` すれば、UI・描画ロジック・ファイル名生成・`package-lock.json`(D11分)はそれぞれ独立に切り戻せる。

### スパイク調査結果(タスク1.1/1.2、`family-chart@0.9.0` のソース調査)

`node_modules/family-chart/dist/family-chart.esm.js` を直接調査した結果、当初想定していなかった公式APIが存在することが判明し、D1・D5を上記の内容へ更新した。

- **`chart.setSortChildrenFunction(fn)` / `chart.setSortSpousesFunction(fn)`**: 子・配偶者の並び順を指定できる公式フック(`CalculateTree` クラスのメソッド、L5321/L5341)。ただしライブラリは指定した関数を適用した**あと**に必ず内部関数 `sortChildrenWithSpouses`(L4-16)を実行し、これは親カードの `data.gender === 'M'` かどうかで比較の昇順/降順を反転させる。半きょうだいがいない単婚のケースでは比較キーが同値になり安定ソートで指定順が保たれるが、半きょうだいが混在する複数婚のケースでは婚姻単位の再グルーピングが優先される。**この `gender` 依存の反転処理が、「性別未設定の人物をクリックすると並び順が入れ替わる」バグの直接原因である**(D1参照)。
- **`chart.setShowSiblingsOfMain(bool)`**(L5215)・**`chart.setAncestryDepth(n)` / `chart.setProgenyDepth(n)`**(L5350/5359): 祖先・子孫の表示世代数と、`main_id` 本人の兄弟姉妹表示を制御する公式フック。`ancestry_depth`/`progeny_depth` はデフォルトで `undefined`(無制限)、`show_siblings_of_main` はデフォルト `false`。全体表示モードは「`show_siblings_of_main: true` を常時有効化 + `main_id` の固定(クリックのたびに動かさない)」の組み合わせで実現する(D5参照)。「走査によらない任意ノード集合の描画」に相当するAPIは存在しないため、非連結クラスタを横断した完全な全体表示は引き続きスコープ外とする。
- 非表示人数バッジ(D6)に相当する機能はfamily-chart側になく、自前実装(BFS)が引き続き必要。この結論に変更はない。
