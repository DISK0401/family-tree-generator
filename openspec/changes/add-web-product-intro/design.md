# Design: Webトップページの製品紹介(ランディングページ)

## Context

- 現在は `/` に直接エディタ(`App.tsx`)がマウントされており、ルーティングは存在しない(`main.tsx` が `App` を直接renderする単一画面)。
- ホスティングはCloudflare Workers static assets。`wrangler.jsonc` の `not_found_handling: "single-page-application"` により、任意のパスで `index.html` が返る構成のため、クライアント側でパスに応じた出し分けが可能。
- dev環境はWorker(`worker/index.ts`)が `X-Robots-Tag: noindex` を付与済み。本番はインデックス可能。
- デザイントークン(`src/styles/tokens.css`)に「墨と紙」の世界観(墨/楮紙/藍/朱、明朝+ゴシック、ライト/ダーク両対応)が定義済みで、コンポーネントはトークン経由で色・書体を参照する規約がある。
- BtoC向けであるため、画像・ビジュアルを用いたモダンな第一印象が求められる(proposal参照)。

## Goals / Non-Goals

**Goals:**

- `/` で製品紹介ページ、`/app` でエディタを表示する画面分離を、最小の複雑さで実現する
- 提供中機能とComing Soon機能が一目で区別できる、ビジュアル主体のモダンなランディングを作る
- 偉人家系図サンプル(複数パターン)を図版で見せ、ワンクリックでエディタに展開して「完成イメージ→実体験」の流れを作る
- 既存の「墨と紙」デザイン言語・トークンを踏襲し、ランディングとエディタでブランドの一貫性を保つ
- ランディング表示時にエディタ用の重い依存(family-chart/D3)を読み込まない
- OGP・meta整備により、SNS共有・検索流入時の見え方を整える

**Non-Goals:**

- 有償機能(戸籍OCR・クラウド保存・共有・課金)の実装。掲載は「Coming Soon」の紹介のみ
- 料金ページ・利用規約・プライバシーポリシー等の独立ページ追加(将来変更で扱う)
- アクセス解析・広告タグの導入(データ送信ゼロの方針を維持し、本変更では入れない)
- 既存エディタのUI変更(エディタ側はサンプル読み込み経路の追加と配置URLの変更のみ。編集機能・データモデルには手を入れない)
- 実在の存命人物の系譜サンプル掲載(個人情報保護法の対象であり本人同意なしの掲載はリスク)、および皇室系譜の掲載(故人のみなら法的には可能だが、商用宣伝への利用は批判リスクがあり避ける)

## Decisions

### D1. ルーティング: ルータライブラリを導入せず、`main.tsx` でのパス判定 + 通常の`<a>`遷移

- `main.tsx` で `window.location.pathname` を判定し、`/app` 配下ならエディタ(`App`)、それ以外(`/` および未知のパス)はランディングをrenderする。
- ランディング→エディタの遷移は通常の `<a href="/app">` によるフルページロードとする。エディタはIndexedDBから状態を復元するため、SPA遷移で保持すべきメモリ上の状態が存在しない。
- 未知のパス(例: `/xyz`)はランディングにフォールバックする(SPA配信のため404ステータスは返せない。ランディングが実質的な案内所を兼ねる)。
- **代替案**: react-router導入(2ルートには過剰。依存とバンドルが増える) / Vite MPA化でHTMLを分ける(ビルド設定・OGP管理が複雑化し、Cloudflareの `single-page-application` 設定とも噛み合わない)。ページ数が増えて遷移が複雑になった時点でルータ導入を再検討する。

### D2. コード分割: ルート単位の `React.lazy` でエディタとランディングを分離

- `main.tsx` から `React.lazy(() => import(...))` + `Suspense` で各画面を遅延読み込みし、ランディング初回表示時にfamily-chart/D3/zustand/idb等のエディタ依存チャンクを読み込ませない。初訪問者の体感速度(BtoCの第一印象)を優先する。
- フォールバックUIは背景色のみの軽量プレースホルダとし、フラッシュを避ける。

### D3. ビジュアル: 写実的スクリーンショットではなく、トークン準拠のSVGイラストを自作する

- ヒーローの主ビジュアルは、家系図(架空の人名・数世代)を「墨と紙」の意匠で描いたインラインSVGイラストとする。色はCSSカスタムプロパティ(`--ink`/`--paper`/`--shu` 等)を参照させ、ライト/ダークテーマに自動追従させる。
- 機能紹介セクションの図版も同様に小さなSVG(和暦入力・自動整列・入出力などのモチーフ)で統一する。
- **代替案**: 実画面のスクリーンショットPNG(テーマ非追従・高DPI対応で肥大・UI変更のたびに陳腐化する) / 外部ストックフォト(世界観が壊れる。外部配信は行わない方針とも不整合)。スクリーンショットは将来、実UIが十分に映える段階で差し替えを検討する。
- イラスト内の人名・年号はすべて架空とし、実在人物を想起させないものにする。

### D4. Coming Soon の表現: 提供中と同一カードグリッドに載せ、バッジ+彩度を落とした表現で区別

- 機能カードは「提供中」「Coming Soon」を同じグリッドで並べ、Coming Soonカードには `--warn` 系トークンのバッジ(「実装予定」)を付け、図版・本文の彩度/濃度を落とす(`--ink-faint` 基調)。ロードマップとして期待を見せつつ、現在使える機能との誤認を防ぐ。
- Coming Soonカードの文言は proposal の法務ルールに従う: 「ご自身で取得した戸籍のスキャンから」「AIが下書きを作成し、ご自身で確認・修正」等、取得代行・全自動を想起させない表現をspecで固定する。

### D5. スタイリング: 既存トークンを流用し、ランディング専用CSSを1ファイル追加

- `src/pages/LandingPage.tsx` + `LandingPage.css`(コンポーネント別CSSという既存慣習に従う)。色・書体・余白・角丸はすべて `tokens.css` のトークンを参照する。
- ランディング特有の大型タイポグラフィが必要な場合は、tokens.cssにスケール追加(例: `--text-3xl`)として拡張し、ハードコードしない。
- レスポンシブはモバイルファースト。カードグリッドは `grid` + `auto-fit/minmax` で列数を流動化し、ヒーローは縦積み(モバイル)→横並び(768px〜)に切り替える。
- `prefers-reduced-motion` 対応は既存のグローバルCSSがカバーする。装飾的なアニメーションを入れる場合もこの規約内で行う。

### D6. HTML head / OGP: `index.html` に静的メタを直書き

- `index.html` に `<meta name="description">`、OGP(`og:title` / `og:description` / `og:image` / `og:type`)、`twitter:card` を静的に追加する。ページが実質1種(ランディング)+アプリのため、動的メタ生成は不要。
- `og:image` 用の静的画像(1200×630 PNG)を `public/ogp.png` として用意する(D3のイラストをベースに書き出し)。
- `og:url` / `og:image` の絶対URLは本番ドメインで記述する。dev環境は既存のWorkerによる `noindex` が効いているため、検索エンジン側の混乱はない。

### D7. データフロー: ランディングは完全静的。送信ゼロを維持

- ランディングページはフォーム・API呼び出し・外部リソース読み込み(フォントCDN・解析タグ含む)を一切持たない。データは「Cloudflareから静的アセットを受信する」方向のみで、クライアントからの送信は発生しない。
- エディタ側の既存データフロー(IndexedDBのみ、サーバ送信なし)には変更を加えない。Supabase/Gemini等の外部APIは本変更では一切登場しない。

### D8. サンプルギャラリー: 静的SVG図版 + `/app?sample=<id>` でエディタに展開

- **人選とパターン**: 徳川家康(側室・多数の子女 = 複数配偶者)、夏目漱石(養子縁組)、渋沢栄一(再婚)、架空の現代家族(典型的な利用イメージ。「架空の家族です」と明記)の4サンプル。偉人はすべて故人で、系譜はWikipedia等で公知の範囲に限定する(死者情報は個人情報保護法の対象外、系譜という事実に著作権は生じない)。各サンプルは主要人物数名〜十数名に絞り、正確性の論争を避けるため簡略化した旨と出典の注記を添える。
- **ランディング側の表示**: 各サンプルの図版は事前描画した静的SVG(D3・トークン準拠)とし、タブまたはカード切替で複数パターンを見せる。family-chartをランディングで読み込まない方針(D2)を維持する。
- **サンプルデータの形式**: エディタの内部ドキュメント形式(`TreeDocument`)のデータを `src/` 内にモジュールとして同梱し、エディタ側チャンクからのみ動的importする。ネットワーク送信は発生しない(静的アセットの受信のみ)。
- **エディタへの展開**: 「このサンプルをエディタで開く」リンクを `/app?sample=<id>` とし、エディタ起動時にクエリを検出してサンプルを読み込む。
  - 既存データがある場合(personCount > 0)は、既存の `ImportExportControl` と同じ「上書き前の確認」パターンを必須とする(黙って `replace` しない)。確認が拒否されたら既存データのまま開く。
  - 読み込み(または拒否)後は `history.replaceState` でクエリを除去し、リロードによる再確認ループを防ぐ。
  - 読み込んだサンプルは通常のドキュメントとして自動保存・編集・エクスポートの対象になる(特別扱いの「読み取り専用モード」は作らない。既存機構の再利用でスコープを最小に保つ)。
- **代替案**: ランディング上でfamily-chartによるライブ描画(D2に反する。バンドル肥大) / エディタに読み取り専用のサンプルモードを新設(状態管理・保存系への影響が大きい割に、確認ダイアログで十分ユーザーデータを守れる)。

## Risks / Trade-offs

- [`/` の役割変更で既存利用者が迷う] → ランディングのヒーローCTA「無料ではじめる」と、ヘッダー常設の「アプリを開く」リンクの二重導線で `/app` へ誘導。IndexedDBデータはオリジン単位の保存でありパス変更の影響を受けない旨をspecの検証シナリオに含める。
- [手作りパス判定が将来のページ増加に耐えられない] → 判定ロジックを `main.tsx` の1箇所に閉じ込め、ページが3つ以上になった時点でルータ導入を検討する(その際も `/app` のURLは維持)。
- [SVGイラストの制作コストと品質のばらつき] → モチーフを家系図・帳簿の線画に限定し、トークン色のみで構成することで作画の自由度を意図的に狭め、品質を安定させる。
- [未知パスがすべてランディングになり、リンク切れに気づきにくい] → 現状ページ数が少なく実害は限定的。ページ増加時に404表示の導入を再検討する。
- [OGP画像がテーマ非追従・手動更新] → OGPはSNSクローラ向けの1枚絵であり、テーマ追従は不要と割り切る。デザイン刷新時の更新をREADMEの運用注意に記載する。
- [サンプル読み込みが既存の家系図データを消してしまう] → 既存データがある場合は上書き前の確認を必須とし、拒否時は既存データを保持したままエディタを開く。specの検証シナリオに含める。
- [偉人系譜の記載内容(人物・続柄)への正確性の指摘] → 主要人物のみに簡略化し、「Wikipedia等の公知情報を基に簡略化したサンプル」である旨を図版・サンプル内に注記する。指摘があれば個別に修正する。
- [歴史上の人物の扱いに対する感情的な反発] → 敬称や表現に配慮し、故人・公知情報のみで構成する。皇室系譜・存命人物は扱わない方針を維持する。

## Migration Plan

1. ランディング実装 + ルート分岐を同一ブランチで実装し、ローカル(`npm run dev`)で `/` と `/app` の出し分け・既存データの継続表示・サンプル読み込み(空の状態/既存データありの両方)を確認する。
2. dev環境へデプロイし、`/app` 直アクセス(SPAフォールバック)・`/app?sample=<id>` 直アクセス・OGPメタ・レスポンシブ表示・noindexヘッダを検証する。
3. 本番デプロイ。デプロイ後に `/` がランディング、`/app` がエディタとして機能すること、既存利用者のIndexedDBデータが `/app` で表示されることを確認する。
4. **ロールバック**: `main.tsx` の分岐を外して `App` を直接renderする1コミットのrevertで従来状態(`/` = エディタ)に完全復帰できる。データ移行を伴わないため、ロールバックにデータ操作は不要。

## Open Questions

- OGP画像(`public/ogp.png`)の最終ビジュアルは実装時にSVGイラストから書き出して確定する(ブロッカーではない)。
- サービス正式名称・キャッチコピーの最終文言はspecでプレースホルダを定め、実装時にユーザーへ確認する。
