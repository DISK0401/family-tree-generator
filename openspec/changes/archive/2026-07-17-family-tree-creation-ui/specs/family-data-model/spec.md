# family-data-model

家系図の内部データモデル。人物(Person / GEDCOM INDI相当)と家族=婚姻単位(Family / GEDCOM FAM相当)の2レコード構造を持ち、複雑な家族関係・和暦・旧字体・不確実な日付を表現する。将来のGEDCOM 7.0/5.5.1入出力への無損失マッピングを前提とした項目定義とする。

## ADDED Requirements

### Requirement: 人物レコードの表現
システムは人物を、構造化された名前(姓・名・ふりがな)、性別(男・女・不明)、出生・死亡イベント(日付+場所)、メモを持つPersonレコードとして表現しなければならない(SHALL)。名前の各フィールドは入力された文字列を正規化せず原文のまま保持しなければならない(MUST)。

#### Scenario: 旧字体・異体字を含む氏名の保持
- **GIVEN** 人物の姓に「髙橋」、名に「廣」を入力した
- **WHEN** Personレコードを保存し再取得する
- **THEN** 「高橋」「広」等の新字体へ正規化されず、入力した字形のまま保持されている

#### Scenario: ふりがなの保持
- **WHEN** 姓「東海林」にふりがな「しょうじ」を設定する
- **THEN** ふりがなは姓・名それぞれに対応づけて保存される(GEDCOMのかな表記へ写像可能な構造)

#### Scenario: 不明な情報の許容
- **WHEN** 名前のみで性別・生没年が不明な人物を作成する
- **THEN** 必須項目は表示名に用いる名前のみで、他のフィールドは未設定のままレコードが成立する

### Requirement: 家族(婚姻単位)の表現
システムは配偶者関係をFamilyレコード(配偶者への参照1〜2件、婚姻・離婚等のイベント、関係種別=法律婚/事実婚/不明)として表現しなければならない(SHALL)。同一人物は複数のFamilyに属することができ、再婚・複数配偶者を表現できなければならない(MUST)。Familyのイベントは時系列のリストとして複数保持できなければならず(MUST)、同一カップルの「結婚→離婚→再婚(復縁)」を1つのFamilyで表現できなければならない(MUST)。

#### Scenario: 再婚の表現
- **GIVEN** 人物Aが人物Bと離婚後、人物Cと再婚した
- **WHEN** A-BのFamily(離婚イベントあり)とA-CのFamilyを作成する
- **THEN** 両方のFamilyが並存し、Aの配偶者履歴として2件の婚姻が取得できる

#### Scenario: 同一相手との復縁の表現
- **GIVEN** 人物Aと人物BのFamilyに婚姻・離婚イベントが記録されている
- **WHEN** 同じFamilyへ2度目の婚姻イベントを追加する
- **THEN** イベントが「婚姻→離婚→婚姻」の時系列リストとして保持され、日付順に取得できる

#### Scenario: 事実婚の表現
- **WHEN** 関係種別を「事実婚」としてFamilyを作成する
- **THEN** 婚姻イベントなしでも配偶者関係として成立し、種別が保持される

#### Scenario: ひとり親の家族
- **WHEN** 配偶者が1人しか判明していない親子関係を登録する
- **THEN** 配偶者参照が1件のみのFamilyが成立し、子を帰属させられる

### Requirement: 子の帰属と続柄種別
システムは子をFamilyへの子リンクとして帰属させ、各リンクに続柄種別(実子・養子・継子・里子・不明)を保持しなければならない(SHALL)。続柄種別はGEDCOMのPEDIへ写像可能でなければならない(MUST)。

#### Scenario: 養子縁組の表現
- **GIVEN** 人物Dが実親のFamily F1に実子として属している
- **WHEN** 養親のFamily F2へ続柄「養子」で子リンクを追加する
- **THEN** DはF1(実子)とF2(養子)の両方に属し、それぞれの続柄種別が区別して取得できる

### Requirement: 不確実な日付の表現
システムは日付をFuzzyDate型(原文表記、グレゴリオ暦の年・月・日=部分欠落可、修飾子=exact/about/before/after/between)で表現しなければならない(SHALL)。修飾子はGEDCOMのDATE修飾子(ABT/BEF/AFT/BET)へ写像可能でなければならない(MUST)。

#### Scenario: 「頃」つき日付の保持
- **WHEN** 出生日に「昭和10年頃」を入力する
- **THEN** 原文「昭和10年頃」、西暦1935年、修飾子aboutとして保存される

#### Scenario: 年のみの部分日付
- **WHEN** 死亡日に年のみ「1988」を入力する
- **THEN** 月・日が欠落した日付として成立し、表示時も「1988年」とだけ表示される

### Requirement: 和暦と西暦の相互変換
システムは明治から令和までの元号について、和暦⇄西暦の双方向変換を提供しなければならない(SHALL)。変換テーブルは改元境界日を含み、データ行の追加のみで古い元号へ拡張できる構造でなければならない(MUST)。存在しない和暦日付は変換エラーとして報告しなければならない(MUST)。

#### Scenario: 和暦から西暦への変換
- **WHEN** 「昭和39年10月10日」を変換する
- **THEN** 1964年10月10日が得られる

#### Scenario: 改元境界日の変換
- **WHEN** 「平成31年4月30日」と「令和元年5月1日」をそれぞれ変換する
- **THEN** 2019年4月30日・2019年5月1日が得られ、逆変換でそれぞれ元の元号表記に戻る

#### Scenario: 存在しない和暦日付の拒否
- **WHEN** 「昭和65年1月1日」(昭和は64年まで)を変換しようとする
- **THEN** 変換エラーが返り、どの元号の範囲外かが分かるメッセージを含む

### Requirement: GEDCOMマッピングを前提とした項目定義
データモデルの全項目は、GEDCOM 7.0のタグ(INDI/FAM/PEDI/BIRT/DEAT/MARR/DIV/DATE等)への対応関係を文書化した対応表を持たなければならない(SHALL)。GEDCOM標準タグへ直接対応しない日本固有情報(和暦原文等)は、拡張タグへ退避可能な明示フィールドとして保持しなければならない(MUST)。

#### Scenario: マッピング対応表の存在
- **WHEN** データモデルの型定義をレビューする
- **THEN** 各フィールドとGEDCOM 7.0タグ(または拡張タグ)の対応表がリポジトリ内ドキュメントとして存在し、対応のないフィールドがない

### Requirement: 保存単位とスキーマバージョン
システムは家系図全体をTreeDocument(全Person・Family、家系図タイトル、更新日時、schemaVersion)として束ねなければならない(SHALL)。schemaVersionは正の整数で、モデル構造の変更時に必ずインクリメントしなければならない(MUST)。

#### Scenario: スキーマバージョンの付与
- **WHEN** 新規にTreeDocumentを作成する
- **THEN** 現行のschemaVersionが記録されている
