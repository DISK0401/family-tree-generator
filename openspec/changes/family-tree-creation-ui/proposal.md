# Proposal: family-tree-creation-ui

## Why

本サービスの根幹は「手作業で家系図を気持ちよく作れる」体験であり、これがなければ無料版のUX仮説検証(段階1 MVP)も、有償版の戸籍自動生成(AI抽出結果の人間校正UI)も成立しない。現状リポジトリには開発基盤(雛形・デプロイ)しか存在せず、アプリケーション本体が空であるため、最初の機能として家系図の手動作成画面を構築する。

競合調査(docs/research/2026-07-market-ux-legal.md)で判明したユーザー不満 — ①人物追加のたびに線・レイアウトが崩れる ②和暦入力が面倒 ③養子・再婚・複数配偶者が描きにくい ④操作説明不足 — を最初から解消する設計とし、「UXで戦う」という差別化戦略の中核を実装する。

## What Changes

- **家系図データモデルの新設**: 人物(個人)と家族関係(婚姻・親子)を表現する内部データ構造を定義する。養子縁組・再婚・複数配偶者・事実婚を表現可能とし、将来のGEDCOM 7.0入出力(別変更で実装)へマッピング可能な構造とする。氏名は旧字体・異体字をそのまま保持し、生没年月日は和暦⇄西暦の相互変換に対応する。
- **家系図編集画面の新設**: 人物の追加・編集・削除、関係(配偶者・子・親・養子)の追加・変更・削除を、家系図キャンバス上から直接行えるUIを実装する。人物編集フォームでは和暦⇄西暦の自動相互変換つき日付入力を提供する。
- **家系図の描画と自動レイアウト**: family-chart (D3.js, MIT) を導入し、人物・関係の変更に追従して自動整列される「崩れないレイアウト」を実現する。ズーム・パン・人物選択などのキャンバス操作を含む。
- **ローカル自動保存**: 編集内容をIndexedDBへ自動保存し、ブラウザ再訪時に復元する。データの全削除機能も併せて提供する(端末上の個人情報をユーザー自身が消せること)。
- **独自のビジュアルデザインの確立**: 本画面はサービスの顔となるため、design フェーズで「ありきたりな生成AI的UI(汎用的なグラデーション・カード羅列)」を避けた独自のデザイン言語(和のトーン・タイポグラフィ・配色・余白設計)を定義し、以降の画面の基準とする。

破壊的変更: なし(新規機能のみ)。

### スコープ外(後続の変更で扱う)

- GEDCOM/JSONのエクスポート・インポート
- PNG/PDF・巻物レイアウトなどの出力
- チュートリアル・オンボーディングの本格実装(本変更では空状態からの導線など最小限の配慮に留める)
- 有償版機能(戸籍スキャン・クラウド保存・共有・課金)

### 無料版 / 有償版への影響

- 対象は**無料版**(クライアント完結)。ただしデータモデルと編集UIは、有償版の「AI抽出結果の人間校正」画面でもそのまま土台となる共通資産。

### 競合UXとの比較・改善

| 競合の不満点 | 本変更での対応 |
|---|---|
| 人物追加で線・レイアウトが崩れる(Excel・買い切りソフト) | family-chartによる自動整列。手動での線引き直しを一切不要にする |
| 西暦入力が面倒(みんなの家系図) | 和暦⇄西暦の自動相互変換つき日付入力 |
| 養子・再婚・複数配偶者が描きにくい(全競合共通) | データモデルの第一級要素として対応し、UIからも直接編集可能にする |
| 旧字体の表示制限(みんなの家系図) | 入力文字列を正規化せずそのまま保持・表示 |
| 公開・遠縁検索前提の設計への不安(みんなの家系図) | サーバ送信ゼロのクライアント完結。プライバシーファーストを明確な対抗軸にする |
| モダンUIの新興ツール(xGrapher) | 汎用モダンUIではなく、家系図・系譜という題材に根ざした独自デザイン言語で差別化 |

### プライバシー・法務への影響

- サーバへのデータ送信は一切発生しない(静的ホスティングのまま)ため、法的義務の変化はなし。
- ただし本変更で初めて**実在の個人情報(氏名・生年月日・続柄)が利用者の端末(IndexedDB)に保存される**。共有端末での閲覧リスクに備え、データ全削除機能を本変更のスコープに含める。保存先がユーザーの端末のみであることをUI上でも明示する。
- 家系図は「観賞用・記念用」の位置づけを維持し、相続関係説明図等の事実証明文書の体裁は採らない。

### ロールバック方針

- 全変更はフロントエンドコードのみで git 管理下にあり、`git revert` + 再デプロイで即時に巻き戻せる(develop→dev環境で検証後にmainへ)。
- IndexedDBのスキーマにはバージョン番号を付与し、将来の変更はマイグレーションで前方互換を保つ。ロールバック時に旧コードが新スキーマを破壊しないよう、スキーマバージョン不一致時は読み取りを中止して警告する設計とする。

## Capabilities

### New Capabilities

- `family-data-model`: 人物・家族関係(婚姻・親子・養子)の内部データモデル。複雑な家族関係の表現、和暦⇄西暦変換、旧字体保持、IndexedDB保存を前提としたスキーマバージョニングを含む。
- `tree-editor`: 家系図の手動編集UI。人物・関係のCRUD、人物編集フォーム(和暦対応日付入力)、キャンバスからの直接操作、空状態からの作成導線。
- `tree-rendering`: family-chartによる家系図の描画・自動レイアウト・キャンバス操作(ズーム・パン・選択)。
- `local-autosave`: IndexedDBへの自動保存・復元・全削除。サーバ送信ゼロの担保。

### Modified Capabilities

なし(既存specは開発基盤のみで、要件変更はない)。

## Impact

- **family-tree-generator リポジトリ**:
  - 新規: 家系図データモデル・ストア(`src/` 配下)、編集画面コンポーネント一式、family-chart連携レイヤ、IndexedDB永続化レイヤ、デザイントークン(配色・タイポグラフィ)
  - 更新: `App.tsx`(エントリ画面差し替え)、`README.md`(機能説明の追記)
  - 依存追加: `family-chart`、IndexedDBラッパー(design で選定)、和暦変換(自前実装かライブラリかを design で選定)
- **my_infra リポジトリ**: 影響なし
- **外部依存・API**: 追加なし(外部通信は発生しない)
