# Design: family-tree-creation-ui

## Context

リポジトリには開発基盤(Vite/React/TS雛形、Cloudflare Workersへのブランチ連動デプロイ)のみが存在し、アプリ本体は未実装。本変更でサービスの根幹となる「家系図の手動作成画面」を無料版(クライアント完結)として構築する。

前提制約:

- 無料版は家系図データ・個人情報を一切サーバへ送信しない(静的ホスティングのまま)
- データモデルは将来のGEDCOM 7.0/5.5.1入出力への無損失マッピングを前提とする(proposal参照)
- 養子・再婚・複数配偶者を表現できること、和暦⇄西暦・旧字体対応
- 競合の不満(レイアウト崩れ・和暦入力の煩雑さ・複雑な家族関係)を解消するUXが差別化の核
- 有償版の「AI抽出結果の人間校正画面」でも本画面のデータモデル・編集UIをそのまま土台にする

### データフロー(本変更の全体像)

```
[利用者の操作] → [Reactアプリ(ブラウザ内)] → [ドメインモデル(メモリ)] ⇄ [IndexedDB(同一端末)]
```

- **すべてのデータはクライアント(ブラウザ)内で完結する。** サーバ・外部APIへの送信は一切ない。
- フォント・アイコン等のアセットもすべて自リポジトリからセルフホストし、CDN・外部オリジンへのリクエストをゼロにする(「サーバに何も送らない」を通信レベルでも実証可能にする)。
- Supabase・Gemini等の外部サービスは本変更では未使用(RLS設計・API送信最小化の検討対象なし)。

## Goals / Non-Goals

**Goals:**

- 人物・関係(婚姻・親子・養子)をキャンバス上から直接CRUDできる編集画面
- 人物追加・変更に追従する崩れない自動レイアウト(手動の線引き直しゼロ)
- 和暦⇄西暦の自動相互変換つき日付入力と、不確実な日付(「頃」等)の保持
- GEDCOMマッピング可能な項目定義を持つドメインモデルと、IndexedDBへの自動保存・復元・全削除
- 「生成AI的な汎用UI」ではない、家系図という題材に根ざした独自デザイン言語の確立

**Non-Goals:**

- GEDCOM/JSONの入出力機能そのもの(項目定義のみ将来対応)
- PNG/PDF・巻物レイアウト出力、印刷最適化
- チュートリアルの本格実装(空状態からの導線のみ)
- 有償版機能(戸籍スキャン、クラウド保存、共有、課金)
- モバイル最適化の完全対応(タブレット以上を主対象とし、スマホは閲覧可能レベル)

## Decisions

### D1. 描画: family-chart を採用し、アダプタ層で隔離する

- **採用**: family-chart(D3.js製、MIT)。家系図特化の自動レイアウト、複数配偶者(再婚・複婚)対応、ズーム/パン、HTML/SVGカスタムカードを備え、React連携可能なことを確認済み。
- **代替案**: GoJS(genogram対応が最も強いが商用ライセンス有償)、dTree/Cytoscape.js/React Flow(家系図特化でなくレイアウトを自作する必要)、完全自前D3(工数大)。個人開発のコスト・特化度からfamily-chartが最適。
- **重要な設計判断**: family-chartのデータ形式(`{id, data, rels:{parents, spouses, children}}`)には**続柄種別(養子等)や離婚の概念がない**。よって family-chart のデータを正とせず、**独自ドメインモデルを唯一の真実(source of truth)とし、描画直前に変換アダプタで family-chart 形式へ射影する**。編集操作もドメインモデルへのコマンドとして実装し、family-chart付属のEditTree(編集UI)は使わない。これによりレンダラ差し替え(将来の自前D3化)が可能になる。

### D2. ドメインモデル: GEDCOMの INDI/FAM 構造に対応づく2レコード+関係属性

TypeScriptの型として以下を定義する(詳細フィールドはspecsで規定):

- **Person(個人 / INDI相当)**: 構造化名前(姓・名・ふりがな。旧字体・異体字は正規化せず原文保持)、性別、出生/死亡イベント(日付+場所)、メモ。
- **Family(家族=婚姻単位 / FAM相当)**: 配偶者2名(または1名)への参照、婚姻/離婚イベントの時系列リスト(同一相手との復縁=結婚→離婚→再婚を1つのFamilyで表現。GEDCOM 7.0のFAM内イベント繰り返しに対応)、関係種別(法律婚・事実婚)。別の相手との再婚・複数配偶者は「同一人物が複数のFamilyに属する」ことで表現する(family-chartのspouses配列とGEDCOMのFAMどちらにも無損失で射影できる)。
- **子どもの帰属**: FamilyからPersonへの子リンクに**続柄種別(pedigree: 実子・養子・継子・里子・不明)**を持たせる(GEDCOM PEDI相当)。
- **FuzzyDate(不確実日付)**: `{ 原文表記(和暦含む), グレゴリオ暦値(年/月/日、部分欠落可), 修飾子(exact/about/before/after/between) }`。GEDCOMのDATE修飾子(ABT/BEF/AFT/BET)へ写像可能。
- **拡張退避**: GEDCOMに直接対応しない日本固有情報(和暦原文など)は各レコードの明示フィールドに保持し、エクスポート時に拡張タグへ退避できる構造とする。
- **TreeDocument**: 上記全体+スキーマバージョン+メタ情報(家系図タイトル、更新日時)を束ねる保存単位。

### D3. 和暦⇄西暦: 自前の元号テーブルで双方向変換

- 明治〜令和の改元境界日を含む静的テーブルを自前実装する。検算に `Intl.DateTimeFormat('ja-JP-u-ca-japanese')` を利用できるが、**和暦→西暦のパースはIntlでは不可能なため自前実装が必須**。npmの既存ライブラリは小規模・メンテ状況が不安定なものが多く、テーブル+変換関数は小さい(数十行+データ)ため自前が妥当。
- 江戸期以前の元号は、テーブルへ行を足すだけで拡張できるデータ駆動構造にする(初期リリースの収録範囲はOpen Questions)。
- 入力UXは「どちらで入力しても、もう一方が即時表示される」双方向フィールドとする(競合の「西暦入力が面倒」への直接回答)。

### D4. 状態管理と履歴: Zustand + スナップショット方式のundo/redo

- グローバル状態はZustand(軽量・ボイラープレート最小)。ドメイン操作(人物追加・関係変更など)は純関数のコマンドとして実装し、ストアはそれを適用するだけにする。
- 誤操作からの復帰はエディタUXの生命線なので、**undo/redoを初期スコープに含める**。家系図データは高々数百人物と小さいため、操作ごとのTreeDocumentスナップショット(structuredClone)で十分。パッチ方式(immer等)への最適化は必要になってから。
- 代替案: Redux Toolkit(過剰)、useReducerのみ(undo/redo・永続化連携を自作すると結局同等の複雑さ)。

### D5. 永続化: idb + デバウンス自動保存 + スキーマバージョンガード

- IndexedDBラッパーは `idb`(軽量・Promise化のみ)を採用。Dexieはクエリ機能が豊富だが、保存単位がTreeDocument一括のためオーバースペック。
- 変更後500ms〜1sのデバウンスで自動保存。保存状態(保存済み/保存中)をUIに常時表示し、「データは端末内にのみ保存される」ことも同じ場所に明示する。
- TreeDocumentに`schemaVersion`を持たせ、読込時に(a)旧バージョン→マイグレーション適用、(b)未知の新バージョン→**読み取りを中止して警告**(ロールバック後の旧コードが新データを破壊しない)。
- 全削除機能は確認ダイアログ(家系図名の再入力等の明示的確認)つきでIndexedDBをクリアする。

### D6. ビジュアルデザイン言語: 「戸籍・系譜の紙の世界」を現代化する

ありきたりな生成AI風UI(紫グラデのヒーロー、丸角カードの羅列、汎用サンセリフ)を避け、題材固有の造形から組み立てる。方向性は「**帳簿・系譜がもつ墨と紙の世界を、現代のツールとして再構成する**」。

- **縦書きの人名カード**: 人物ノードは `writing-mode: vertical-rl` の縦書きを基本とする(HTMLカスタムカードで実装)。家系図・系譜の伝統的な佇まいを画面の第一印象で成立させる、本デザインの核となる一手。
- **タイポグラフィ**: 人名・見出しは明朝系(ヒラギノ明朝/游明朝/Noto Serif CJK JPのシステムスタック)、UI操作系・フォーム・補助情報はゴシック系(ヒラギノ角ゴ/Noto Sans CJK JPのシステムスタック)の2役構成。数字は `tabular-nums`。旧字体・異体字の字形保持を最優先し、v1はWebフォントを導入しない(=外部リクエストもゼロ、CJKフォントの巨大ペイロードも回避)。
- **配色トークン**: 墨(テキスト)・楮紙(地)・藍(操作・リンク=インタラクティブの印)・朱(選択・印=ここぞの一点)の4役+セマンティック色(警告・削除)。朱は「朱印」のように選択中人物のマーカーや重要アクションのみに絞り、画面を朱だらけにしない。ライト/ダーク両テーマをCSSカスタムプロパティのトークンで定義する(ダークは単純反転でなく「夜の帳簿」= 藍鉄の地に生成りの文字)。
- **系線の造形**: 家系の線(親子・婚姻線)は太さ・実線/破線(養子は破線等)で意味を持たせ、単なるコネクタでなく「系譜の線」として描く。装飾(数字マーカー・絵文字・グラデーション)は使わない。
- **モーション**: レイアウト変化時のD3トランジションのみ(人物追加時に樹が滑らかに組み変わる)。`prefers-reduced-motion` を尊重。
- 具体的なトークン値(hex・タイプスケール)は実装タスクの最初で `src/styles/tokens.css` として定義し、以降の画面すべての基準にする。

### D7. 画面構成: キャンバス主体+サイドパネル編集

- 画面はキャンバス(家系図)を全面に置き、人物選択時に編集パネルが右側から開く構成。モーダルで作業を中断させない。
- 人物カード・選択中人物の近傍に「配偶者を追加」「子を追加」「親を追加」のコンテキストアクションを出し、**フォームから関係を組み立てるのではなく、図の上で家族を育てる**操作モデルにする(競合の「Excelで線を引き直す苦痛」の対極)。
- 空状態(人物ゼロ)では「最初の人物(あなた自身など)を追加」の一点に導線を絞ったガイドを表示する。

## Risks / Trade-offs

- **[family-chartの表現力限界]** 養子の別種リンク描画・離婚の視覚表現はネイティブ機能にない → カスタムカード+リンクへのCSS/SVG後処理で対応。実装冒頭に**スパイクタスク**(複数配偶者・養子・縦書きカードの描画検証)を置き、不可能ならレンダラ自作へ早期に方針転換する(ドメインモデルはD1により無傷)。
  - **スパイク結果(family-chart@0.9.0で検証済み → 採用続行)**: ①複数配偶者は`rels.spouses`複数指定で2本の婚姻線が正しく描画される ②養子リンクは描画後に`path.link`のD3 datum(`source`/`target`)から対象を特定してクラス付与→CSS破線化が可能(`updateTree`のたびに`afterUpdate`フックで再適用する必要あり) ③`setCardInnerHtmlCreator`によるHTMLカスタムカードで`writing-mode: vertical-rl`の縦書き・旧字体(髙・廣)表示が動作。ただしfamily-chart既定CSSとカード寸法(`setCardDim`)の調整が必要(文字詰まりが発生したため実装時にカード設計を詰める)。検証コードは`spike/`に保存。
- **[family-chartがv0.x(枯れていない)]** 破壊的変更・不具合のリスク → バージョン固定(lockfile+セマンティックレンジ禁止)、アダプタ層への隔離で影響範囲を限定。
- **[縦書きの実装難]** カード内の縦書きとレイアウト計算が干渉する可能性 → スパイクで検証し、崩れる場合は横書きカードへフォールバック(トークン・型構成は変えない)。
  - **6.3実装時に判明した2点の落とし穴(解決済み)**: ①`setCardInnerHtmlCreator`でカード内側を完全に置き換えると、family-chart既定のカードサイズCSS(`.f3 div.card-rect`等、コンテナに`f3`クラスが付与されている前提)が適用されない。マウント先のコンテナに`f3`クラスを付与した上で、`setCardDim`と同じ値をCSSカスタムプロパティ経由で`.tree-card`へ明示指定する必要がある(さもないとfamily-chartの`tree_position:'fit'`のスケール計算も含めて崩れる)。②`writing-mode: vertical-rl`を指定した要素をflexコンテナの中でさらにflexネストすると、ブラウザの直交フロー(orthogonal flow)の高さ計算が崩れテキストが数px角に潰れる。縦書き要素は中間flexラッパーを挟まず、外側flexコンテナの直接の子として配置することで解消した。
- **[IndexedDBのデータ消失]** ブラウザのデータ消去・シークレットモードで家系図が消える → エクスポート機能実装(後続変更)までの間、保存表示部で「端末内保存であること」と消失リスクを明示。`navigator.storage.persist()` で永続化を要求する。
- **[和暦変換の境界誤り]** 改元日(例: 平成31年4月30日/令和元年5月1日)前後の誤変換 → 境界日を含む変換テーブルの単体テストを必須とする。
- **[スナップショットundoのメモリ増]** 巨大家系図で履歴が肥大 → 履歴上限(例: 100件)を設ける。それでも問題になればパッチ方式へ移行。
- **[システムフォント依存]** OSにより明朝の字形・旧字体グリフが揺れる → 許容する(v1)。品質が問題になればサブセット化したWebフォントのセルフホストを後続変更で検討。

## Migration Plan

1. featureブランチで実装 → `develop` マージで dev 環境へ自動デプロイ、実機確認
2. 問題なければ `main` マージで本番反映
3. ロールバック: `git revert` + 再デプロイ(クライアント完結のためサーバ側の巻き戻し作業なし)。IndexedDBはD5のスキーマバージョンガードにより、旧コードが新スキーマデータを破壊しない
4. 本変更は新規画面のみで既存機能への影響なし(既存はプレースホルダのApp.tsxのみ)
